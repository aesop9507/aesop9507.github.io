## 왜 지금 First Action을 다시 봐야 하나

최근 백엔드 조직에서 장애 대응의 핵심은 단순히 **빨리 인지하는 것**을 넘어, 인지 직후 **의미 있는 첫 조치(First Action)** 를 얼마나 빠르게 실행하느냐로 이동하고 있습니다.

우아한형제들 기술블로그의 최신 Backend 글(2026-02-03)에서도 같은 결론이 나옵니다.
핵심은 명확합니다.

- 탐지 속도만 빠른 대응은 고객 영향 시간을 충분히 줄이지 못한다.
- 핫픽스 중심 대응은 원인 분석 + 코드 수정 + 재배포 시간이 누적되어 장기화되기 쉽다.
- 롤백/스케일 조정 같은 **기계적 완화 조치**를 우선 실행하면 고객 영향 면적을 빠르게 줄일 수 있다.

이 글은 해당 사례를 바탕으로, 백엔드 팀이 실무에 바로 적용할 수 있는 형태로 First Action 운영 프레임을 재구성한 내용입니다.

---

## 1) First Action의 정의를 코드처럼 명확히 하자

운영에서 가장 먼저 실패하는 지점은 “첫 조치”의 정의가 팀마다 다르다는 점입니다.

실무에서 권장하는 정의는 다음과 같습니다.

> First Action = 장애 인지 이후, 추가 분석 없이 즉시 실행 가능한 사전 정의 완화 조치의 최초 실행 시점

여기서 중요한 것은 “정답 조치”가 아니라 “**고객 영향 축소 속도**”입니다.

### 권장되는 First Action 후보

- 직전 안정 버전으로 롤백
- 문제 인스턴스/파드 격리
- 트래픽 우회(라우팅 비율 조정)
- 임시 Rate Limit 강화
- 스케일 아웃/큐 드레인 가속

핫픽스 자체가 나쁜 것은 아니지만, **First Action 이전에 핫픽스만 붙잡는 패턴**은 대체로 복구 시간을 늘립니다.

---

## 2) 장애 라이프사이클을 지표화해야 개선이 시작된다

우아한형제들 사례에서 인상적이었던 부분은 First Action을 감각이 아니라 **관찰 가능한 메트릭**으로 끌어올렸다는 점입니다.

백엔드 팀에서 최소로 가져가야 할 타임라인은 다음 4개입니다.

1. `T_detect` : 탐지 시점
2. `T_ack` : 온콜 인지/접수 시점
3. `T_first_action` : 첫 완화 조치 실행 시점
4. `T_mitigated` : 고객 영향 실질 완화 시점

그리고 아래 파생 지표를 운영 대시보드에 고정합니다.

- **TTFA (Time To First Action)** = `T_first_action - T_detect`
- **TTM (Time To Mitigate)** = `T_mitigated - T_detect`
- **Hotfix-first 비율** = First Action 없이 핫픽스로 시작한 케이스 비율

이 세 지표만 안정화해도 “인지가 빨랐는데 왜 길게 아팠는가?”를 설명할 수 있게 됩니다.

---

## 3) Runbook는 ‘판단 문서’가 아니라 ‘실행 문서’여야 한다

대부분 팀의 런북은 원인 분석 중심입니다.
하지만 First Action 관점에서는 런북의 첫 화면이 다음 질문에 답해야 합니다.

- 지금 바로 되돌릴 수 있는가?
- 트래픽을 줄이거나 우회할 수 있는가?
- 위험한 쓰기 경로를 일시 차단할 수 있는가?

즉, 런북 초반은 “왜”가 아니라 “**지금 뭘 누를지**”로 구성해야 합니다.

### 추천 런북 템플릿 (요약)

- 증상 분류: 오류율 급증 / 지연 급증 / 데이터 불일치
- 즉시 조치 버튼: 롤백 / 우회 / 격리 / 스케일
- 중단 조건: 고객 영향 완화 여부
- 이후 단계: 원인 분석, 핫픽스, 재발 방지

---

## 4) 백엔드 아키텍처에서 First Action 성공률을 높이는 설계

First Action은 운영 역량만이 아니라 아키텍처 결정의 결과물입니다.

### (1) 안전한 롤백 경로를 항상 유지

- DB 마이그레이션은 backward-compatible 우선
- feature flag로 기능 노출 제어
- 배포 단위는 작게, 영향 범위는 좁게

### (2) 격리 가능한 경계 설계

- 쓰기 경로와 조회 경로 분리
- 외부 연동 장애 시 circuit breaker + fallback 기본화
- 테넌트/파트너 단위 트래픽 격리 스위치 제공

### (3) 관측성의 목표를 바꾸기

관측성의 목표는 “정확한 원인 규명”만이 아닙니다.
장애 초기에는 “어떤 First Action이 가장 효과적인지 빠르게 결정”할 수 있어야 합니다.

따라서 대시보드는 아래 기준으로 재편해야 합니다.

- 서비스 단위보다 **고객 영향 단위** 우선
- 평균값보다 p95/p99, 오류 버짓 소진 속도 우선
- 탐지 알람보다 “권장 첫 조치 힌트” 노출

---

## 5) 바로 적용 가능한 2주 실행 플랜

### Week 1: 정의와 계측
- 최근 장애 10건에서 `T_first_action` 역추적
- TTFA/TTM 기준선 산출
- 장애 분류별 First Action 후보 1~2개 확정

### Week 2: 자동화와 훈련
- 온콜 봇/런북에 First Action 단축 명령 연결
- 게임데이에서 “핫픽스 금지, 완화 우선” 시나리오 리허설
- 회고 템플릿에 TTFA, 조치 타입(롤백/핫픽스) 필수 입력

2주만 실행해도 “감으로 잘 대응했다”에서 “지표로 빨라졌다”로 전환됩니다.

---

## 마무리: 장애 대응의 품질은 첫 10분에 결정된다

백엔드 장애 대응은 기술력 경쟁이기도 하지만, 결국 **운영 설계 경쟁**입니다.
First Action을 조직의 공통 언어로 만들면 다음 변화가 나타납니다.

- 장애 초기에 불필요한 토론이 줄어든다.
- 고객 영향 시간을 구조적으로 단축할 수 있다.
- 핫픽스 의존도가 낮아지고 재현 가능한 대응이 늘어난다.

핵심은 단순합니다.

> “원인을 빨리 맞히는 팀”보다 “영향을 먼저 줄이는 팀”이 더 강한 운영팀입니다.
