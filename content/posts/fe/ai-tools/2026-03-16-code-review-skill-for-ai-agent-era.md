# 코드 리뷰 역량이 AI 에이전트 활용도를 결정한다

이번 모니터링에서 가장 인상 깊었던 주제는 KOFE Article의 **「코드 리뷰를 잘한다면, AI 에이전트를 잘 활용하게 될 것입니다」**였다.  
핵심은 단순하다.

> AI를 잘 쓰는 능력은 프롬프트 문장력이 아니라, **리뷰를 통해 방향을 교정하는 설계 능력**에 가깝다.

프론트엔드 팀 맥락에서 이 메시지는 특히 중요하다. FE 코드는 사용자 경험, 접근성, 성능, 상태 일관성까지 동시에 만족해야 해서 “동작하는 코드”와 “운영 가능한 코드”의 간극이 크기 때문이다.

---

## 1) 왜 "코드 생성"보다 "코드 리뷰"가 더 중요해졌나

AI 에이전트는 빠르게 구현한다. 문제는 속도가 아니라 **오차 방향**이다.

- 같은 기능이라도 상태 모델이 잘못되면 릴리즈 후 결함 밀도가 급증한다.
- 접근성·성능·관측성은 프롬프트 한 번으로 자동 완결되지 않는다.
- "대충 맞는 코드"가 빠르게 많이 쌓이면, 기술 부채는 선형이 아니라 지수적으로 증가한다.

즉, 생산성의 병목은 "작성 속도"에서 "검증 품질"로 이동했다.

---

## 2) AI 에이전트를 주니어 개발자로 보면 운영이 쉬워진다

실무적으로는 AI를 다음처럼 취급하는 것이 가장 안전하다.

- **장점:** 구현 속도, 대안 제시, 반복 작업 자동화
- **약점:** 경계 조건 누락, 암묵적 도메인 규칙 미반영, 과한 추상화

그래서 리드 개발자의 역할은 "직접 타이핑"이 아니라 **리뷰를 통한 설계 통제**가 된다.

### 리뷰어의 새 책임

1. 문제 정의가 코드에 반영됐는지 확인
2. 경계 조건(실패/지연/중복요청)을 강제
3. 장기 유지보수성(읽기 난이도/변경 비용) 평가
4. 팀 표준(접근성, 로깅, 테스트 계약) 위반 차단

---

## 3) 프론트엔드에서 꼭 보는 AI 코드 리뷰 4축

### A. 상태 모델링
- 상태가 불명확한 `boolean` 난립인지
- 비동기 상태(`idle/loading/success/error`)가 분리되어 있는지
- UI 상태와 서버 상태의 책임 경계가 명확한지

### B. 부수효과 경계
- `useEffect`가 비즈니스 로직을 과도하게 포함하는지
- 이벤트 핸들러/액션 계층으로 이동 가능한지
- 중복 fetch, stale closure, race condition 위험이 있는지

### C. 사용자 체감 성능
- 초기 렌더 비용이 커지는 구조인지
- 리스트/차트에서 메모이제이션 또는 가상화가 필요한지
- 로딩/에러/빈 상태가 UX 관점에서 일관적인지

### D. 운영성(Observability)
- 장애 시 원인 추적 가능한 로그 키(requestId, feature, action)가 있는지
- 에러 메시지가 사용자용/운영자용으로 분리되어 있는지
- 회귀 방지 테스트가 함께 추가됐는지

---

## 4) "좋은 프롬프트"보다 "좋은 리뷰 루프"를 설계하라

AI 협업의 실전 단위는 1회성 생성이 아니라 **반복 루프**다.

### 권장 루프 (FE 팀 기준)

1. **작업 쪼개기**: UI/상태/데이터/테스트로 분리
2. **명시적 제약 전달**: 접근성, 타입 strict, 성능 예산, 테스트 범위
3. **초안 생성**: AI가 구현
4. **리뷰 피드백**: 설계 중심으로 수정 지시
5. **검증 자동화**: lint/type/test/e2e 스모크
6. **머지 전 회고**: 재사용 가능한 프롬프트/리뷰 체크리스트 추출

이 루프가 정착되면, 팀은 "AI에게 일을 시키는 팀"이 아니라 "AI를 훈련 가능한 시스템으로 운영하는 팀"이 된다.

---

## 5) 팀에 바로 적용 가능한 체크리스트

### PR 템플릿에 추가할 항목
- [ ] 상태 전이 다이어그램 또는 표가 있다
- [ ] 실패 시나리오(네트워크/권한/빈 데이터) 반영
- [ ] 접근성(키보드 포커스, aria 라벨) 검증
- [ ] 관측성 필드(event/action/requestId) 반영
- [ ] 회귀 방지 테스트 포함

### 리뷰 코멘트 원칙
- 구현 방식보다 **설계 의도**를 먼저 질문
- "이 코드가 틀렸다"보다 "이 상태 모델이 확장 가능한가"를 논의
- 코멘트는 재사용 가능 규칙으로 남기기 (팀 위키/가이드 업데이트)

---

## 6) 결론: AI 시대의 시니어 역량은 "작성"이 아니라 "판단"

프론트엔드 팀에서 AI 도입 효과를 가르는 기준은 툴 선택이 아니다.

- 얼마나 빠르게 생성하느냐보다
- 얼마나 정확하게 검토하고 교정하느냐가
- 결국 제품 품질과 릴리즈 속도를 함께 결정한다.

그래서 코드 리뷰 역량은 더 이상 "좋은 습관"이 아니라,  
**AI 레버리지를 극대화하는 핵심 운영 기술**이 되었다.
