## 왜 이 주제를 골랐나

오늘 확인한 여러 아티클 중 가장 실무 밀도가 높았던 글은 **`독학 프런트엔드 개발자들을 위한 프로그래밍 원칙`**이었습니다.

핵심이 명확했습니다.

- "좋은 말"(YAGNI, DRY)을 외우는 데서 끝내지 않고,
- **지금 코드에서 바로 적용할 수 있는 실행 규칙**으로 바꿔주며,
- 결과적으로 **개발 속도 + 유지보수성**을 함께 올리는 방식이기 때문입니다.

특히 프론트엔드 팀에서 자주 겪는 문제(성급한 추상화, 과한 범용화, 섞여버린 책임, 읽기 어려운 함수)를 정확히 찌르는 내용이라, 이번 포스팅 주제로 선택했습니다.

---

## 1) 원칙은 "슬로건"이 아니라 "의사결정 도구"여야 한다

실무에서 자주 듣는 말:

- 성급한 최적화는 악의 근원
- YAGNI
- DRY

문제는, 이 문장들만으로는 **지금 무엇을 해야 하는지 결정하기 어렵다**는 점입니다.

예를 들어,

- "미래 확장"을 생각하다가 지금 기능 구현이 지연되거나,
- "중복 제거"를 너무 빨리 하다가 오히려 추상화가 복잡해지는 경우가 생깁니다.

즉, 원칙은 선언이 아니라 **트레이드오프를 고르는 규칙**이어야 합니다.

---

## 2) 3의 법칙: YAGNI와 DRY를 동시에 지키는 가장 현실적인 방법

이번 아티클의 가장 실용적인 포인트는 **Rule of Three(3의 법칙)**입니다.

### 적용 순서

1. **첫 번째**: 그냥 만든다 (현재 요구를 충족하는 가장 단순한 구현)
2. **두 번째**: 필요하면 복붙해서 또 쓴다 (약간의 차이는 허용)
3. **세 번째**: 그때 공통점을 확인하고 추상화한다

### 왜 강력한가

- 너무 일찍 일반화하는 리스크를 줄여줌
- 실제 변형 패턴이 보인 뒤에 추상화하므로 API 품질이 올라감
- "중복 제거"보다 "문제 이해"를 먼저 하게 만듦

프론트엔드 기준으로는,

- 버튼 컴포넌트 variant를 초기에 과도하게 열어두기보다,
- 실제 3개 사용처가 생긴 시점에 props/slot 구조를 정리하는 방식이 안정적입니다.

---

## 3) "작동하게 → 올바르게 → 빠르게": 우선순위를 강제하는 프레임

코드 품질이 흔들리는 큰 이유는, **우선순위 충돌**입니다.

- 아직 안 되는 기능인데 성능 미세최적화부터 하거나,
- 명확한 요구 검증 전 구조 정리부터 시작해 시간을 태웁니다.

여기서 유용한 체크가 바로:

1. **작동하는가?**
2. **올바른가?**
3. **충분히 빠른가?**

이 순서대로만 판단해도, 잘못된 최적화의 70% 이상을 피할 수 있습니다.

### 프론트엔드에서의 실전 적용

- Step 1: 사용자 액션이 실제 완료되는지(기능 성공)
- Step 2: 예외 상태/경계값에서 정확히 동작하는지(정합성)
- Step 3: 렌더링/번들/네트워크 병목을 측정 후 최적화(성능)

핵심은 **측정 없는 성능 최적화는 보류**입니다.

---

## 4) 단일 책임 + 단일 추상화 레벨: "읽히는 코드"의 최소 조건

### 단일 책임 원칙(SRP)

하나의 함수/모듈은 한 가지 이유로만 변경되어야 합니다.

실패 패턴:

- API 호출 + 데이터 변환 + UI 포맷팅 + 에러 토스트를 한 함수에서 처리

개선 패턴:

- fetch 계층 (입출력)
- mapper 계층 (도메인 변환)
- view-model 계층 (화면 전용 가공)

분리하면 변경 영향 범위가 줄고 테스트가 쉬워집니다.

### 단일 추상화 레벨

한 함수 안에서

- 저수준 연산(파싱/인덱스 조작)과
- 고수준 비즈니스 의도(권한 판단/노출 정책)

를 섞지 않는 원칙입니다.

함수 설명에 "그리고"가 많이 붙으면 대부분 레벨이 섞였다는 신호입니다.

## 5) FE 팀에서 바로 써먹는 코드 리뷰 체크리스트

다음 PR 리뷰에서 바로 적용할 수 있는 체크들입니다.

- [ ] 이 추상화는 "세 번째 사용처" 이후에 만들어졌는가?
- [ ] 아직 동작 검증 전인데 구조/최적화부터 하고 있지 않은가?
- [ ] 함수 설명이 한 문장으로 가능한가? (`그리고`가 2번 이상 나오면 분리)
- [ ] 데이터 접근, 도메인 판단, UI 표현이 한 곳에 섞여 있지 않은가?
- [ ] 성능 개선 항목에 실제 측정 근거(LCP, 렌더 횟수, 프로파일링)가 있는가?

---

## 6) 내 해석: "좋은 코드"보다 "좋은 리듬"이 먼저다

이번 글의 진짜 가치가 여기 있다고 봅니다.

좋은 코드는 한 번에 완성하는 결과물이 아니라,

- 과잉 설계를 늦추고,
- 요구 이해를 먼저 확보하고,
- 필요 시점에만 추상화하는,

**좋은 개발 리듬**에서 나온다는 점입니다.

프론트엔드처럼 요구 변경이 잦고 제품 실험이 빠른 환경일수록,
이 리듬이 생산성을 결정합니다.
