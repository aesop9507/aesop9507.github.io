## 왜 이 주제를 골랐나

오늘 모니터링한 후보들 중 가장 실무 전환 가치가 컸던 주제는 KOFE Article의
**"자기가 작업하지 않는 소프트웨어는 설계할 수 없습니다"**였습니다.

프론트엔드 팀에서 아키텍처 결정이 실패하는 대표적인 이유는 기술 역량 부족이 아니라,
**설계자가 실제 사용 맥락(사용자/운영자/개발자)을 충분히 체험하지 못한 상태에서 결정을 내리는 구조**에 있습니다.

이 글은 그 문제를 단순한 슬로건이 아니라, 실행 가능한 운영 원칙으로 바꿔줍니다.

---

## 핵심 주장: 설계 품질은 ‘문서의 정교함’보다 ‘피드백 거리’에 비례한다

좋은 설계는 처음부터 완벽한 설계를 뽑아내는 능력이 아니라,
**잘못된 가정을 빠르게 발견하고 수정하는 피드백 체계**에서 나옵니다.

프론트엔드 관점에서 피드백 거리를 늘리는 요인:

- 기획서/피그마/컴포넌트 명세가 실제 사용자 흐름과 분리됨
- 구현자가 운영 데이터(오류율, 이탈 구간, 사용자 입력 패턴)를 보지 못함
- 설계 리뷰가 코드/실사용 시나리오 없이 추상 토론으로 끝남
- “재사용성”을 과대평가해 도메인 맥락을 지운 공용 컴포넌트를 양산함

결과적으로,
초기엔 그럴듯해 보이지만 실제 제품에서는 불편하고 수정 비용이 큰 구조가 남습니다.

## 프론트엔드에서 자주 발생하는 ‘비사용 기반 설계’의 실패 패턴

### 1) 너무 이른 공통화(Over-generalization)

- 현상: 2~3개 화면에서만 쓰일 로직을 “전사 표준”으로 포장
- 문제: 옵션/플래그가 계속 늘어나고 컴포넌트 API가 비대해짐
- 신호: Boolean prop 4개 이상, 분기 많은 렌더 경로, Storybook만 깔끔하고 실서비스는 난해

### 2) 시각 설계 중심, 상호작용 설계 부재

- 현상: 정적 시안은 훌륭하지만 로딩/오류/권한/경합 상태 정의가 없음
- 문제: 구현 단계에서 임시 분기 추가 → 상태 폭발
- 신호: “일단 성공 케이스 먼저”가 반복되고 장애 시 UX가 즉흥적으로 붙음

### 3) API 계약과 화면 상태 모델의 분리

- 현상: 백엔드 응답 필드에 화면 구조를 억지로 맞춤
- 문제: 작은 스펙 변경이 다수 화면을 연쇄 파손
- 신호: 컴포넌트가 API shape를 직접 참조, selector/adapter 계층 부재

### 4) 운영 체감 없는 성능 최적화

- 현상: 추측 기반 최적화(useMemo 남용, premature split)
- 문제: 코드 복잡성은 증가하지만 체감 성능 개선은 미미
- 신호: LCP/INP/CLS 등 핵심 지표 기준선 없이 “빠른 것 같음”으로 의사결정

## 실행 원칙: “설계자 = 사용자 + 구현자 + 운영자”의 최소 교집합을 확보하라

아래 5가지는 프론트엔드 팀에서 즉시 적용 가능한 원칙입니다.

### 원칙 1) 설계 전, 핵심 시나리오를 직접 ‘작동’시켜 본다

- Figma 리뷰만으로 승인하지 않기
- 클릭 가능한 프로토타입 또는 최소 기능 스파이크를 먼저 만듦
- 정상/오류/지연/권한거부/빈데이터 5개 상태를 강제 점검

**체크 질문**

- 사용자가 3분 안에 목표를 달성하는가?
- 실패 시 복구 경로가 1~2스텝으로 명확한가?
- 모바일/저성능 환경에서도 흐름이 유지되는가?

### 원칙 2) 설계 문서에 ‘불변 조건(Invariant)’을 명시한다

디자인 토큰이나 컴포넌트 사양보다 먼저,
해당 기능이 반드시 지켜야 할 도메인 불변 조건을 선언합니다.

예:

- “결제 확인 전엔 확정 상태를 노출하지 않는다”
- “민감 액션은 항상 명시적 재확인 단계를 거친다”
- “오프라인/재시도 시 동일 요청이 중복 반영되지 않는다”

불변 조건이 있으면 화면 구조가 바뀌어도 핵심 규칙은 흔들리지 않습니다.

### 원칙 3) UI 계층에서 도메인 변환을 분리한다

컴포넌트가 API 응답 원형을 직접 다루지 않게 설계합니다.

- `API DTO -> Domain Model -> View Model` 변환 계층 분리
- 컴포넌트는 View Model 기준으로만 렌더
- API 변경 영향 반경을 adapter 레이어로 한정

이 방식은 리팩터링과 테스트의 경계를 명확히 만들어 장기 유지보수 비용을 줄입니다.

### 원칙 4) ‘배포 후 관찰’을 설계 프로세스에 포함한다

설계 완료의 정의(DoD)를 “코드 머지”가 아니라 “운영 검증 완료”로 확장합니다.

- 필수 이벤트 로깅(전환, 이탈, 에러 타입) 사전 정의
- 배포 후 24~72시간 모니터링 항목 지정
- 예상과 다른 사용자 행동이 나오면 설계 문서를 즉시 업데이트

즉, 설계는 문서가 아니라 **살아 있는 운영 루프**가 됩니다.

### 원칙 5) 재사용은 ‘실사용 검증’을 통과한 뒤에만 승격한다

공용 컴포넌트 승격 기준을 명시적으로 둡니다.

- 최소 3개 이상의 실제 사용처
- 분기 없는 공통 인터페이스 가능
- 접근성/테스트/문서화 완료
- 도메인 특수성 제거 후에도 의미 보존

이 기준 없이 공통화하면 “재사용 가능한 쓰레기”가 쌓이기 쉽습니다.

## React/Next.js 팀을 위한 실전 적용 예시

### 예시: 포트폴리오 대시보드 필터 패널

문제 상황(기존):

- 필터 UI가 API 파라미터 구조와 강결합
- 새 조건 추가 시 컴포넌트/쿼리키/URL 동기화가 동시에 깨짐
- 로딩/에러 상태가 탭마다 다르게 처리되어 UX 불일치 발생

개선 접근:

1. 도메인 필터 모델을 별도 정의 (`PortfolioFilter`)
2. API adapter에서만 파라미터 매핑 수행
3. UI는 모델 기준으로 동작하고 URL 직렬화 모듈을 분리
4. 상태 전이를 state machine처럼 명시(Idle/Loading/Success/Error/Recovering)
5. 배포 후 이탈 이벤트를 측정해 기본값/상호작용 순서 재조정

성과(기대 효과):

- 스펙 변경 시 수정 지점 예측 가능
- 필터 확장 시 회귀 위험 감소
- 사용자 과업 완료율 개선(불필요 클릭 감소)

## 팀 운영 관점의 인사이트

### 1) 설계 리뷰 참석자 구성을 바꿔야 한다

“잘 그린 화면”을 평가하는 회의가 아니라,
**실패 시나리오를 함께 검증하는 회의**가 되어야 합니다.

권장 참여:

- FE 구현 담당
- 도메인 책임자(PO/PM)
- QA 또는 사용자 관찰 가능한 역할

### 2) 기술 부채는 구현보다 설계 단계에서 먼저 발생한다

대부분의 리팩터링 비용은 코드 스타일 문제가 아니라,
초기 상태 모델/경계 설정 오류에서 출발합니다.

### 3) 문서의 목적은 승인 자료가 아니라 학습 기록이다

설계 문서에는 “왜 이 결정을 했는지”와 함께
“어떤 지표가 나오면 이 결정을 철회할지”를 포함해야 합니다.

## 당장 적용 가능한 체크리스트

다음 기능을 시작할 때 아래 7개 중 5개 이상을 만족하지 못하면,
설계를 더 다듬는 것이 구현 속도보다 우선입니다.

- [ ] 핵심 사용자 과업 1개를 3분 내 완료 가능한가?
- [ ] 정상/오류/지연/빈데이터/권한거부 상태를 모두 정의했는가?
- [ ] 도메인 불변 조건을 문서 상단에 명시했는가?
- [ ] UI와 API 모델 사이 변환 계층이 있는가?
- [ ] 배포 후 모니터링 지표(전환/이탈/에러)를 정했는가?
- [ ] 공통화 기준(실사용 3곳 이상)을 통과했는가?
- [ ] 변경 시 영향 범위를 설명할 수 있는가?

## 마무리

“자기가 작업하지 않는 소프트웨어는 설계할 수 없다”는 말은
설계 권한을 좁히자는 뜻이 아닙니다.

오히려 반대입니다.

- 설계자는 구현을 경험하고,
- 구현자는 사용자 맥락을 이해하고,
- 팀은 운영 데이터를 학습해

**설계-구현-운영을 하나의 피드백 루프로 묶어야 한다**는 제안입니다.

프론트엔드에서 좋은 설계는 결국,
아름다운 구조보다 **실제 사용에서 덜 깨지고 빨리 개선되는 구조**입니다.
