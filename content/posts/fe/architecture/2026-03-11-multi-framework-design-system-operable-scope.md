## 배경: 멀티 프레임워크 지원은 ‘기술 문제’보다 ‘운영 문제’다

레거시 Vue와 신규 React/Next.js를 동시에 운영하는 조직에서는 자연스럽게 이런 목표를 세웁니다.

- 공통 디자인 토큰을 만들고
- 공통 컴포넌트를 만들고
- React/Vue를 같은 수준으로 지원하자

문제는 이 목표가 **초기 구축 단계에서는 그럴듯하지만**, 실제 운영 단계에서 비용이 폭발하기 쉽다는 점입니다.

여기어때 기술블로그의 최신 글(2026-03-10)은 이 문제를 매우 현실적으로 보여줍니다.
핵심 메시지는 간단합니다.

> 멀티 프레임워크 디자인 시스템의 성공 조건은 “얼마나 많이 공통화했는가”가 아니라,
> “우리 팀이 장기적으로 감당 가능한 지원 범위를 정의했는가”다.

---

## 처음엔 정답처럼 보였던 구조: Core + Adapter

사례에서 제시한 초기 구조는 아래와 같습니다.

- `core`: 토큰/스타일/헤드리스 로직의 단일 출처
- `react`: React 렌더링 어댑터
- `vue`: Vue 렌더링 어댑터

이 구조의 장점은 분명합니다.

- 비즈니스/상태 로직 재사용
- 프레임워크별 UI 표면 분리
- 설계 의도(규칙 한 곳, 연결은 각 프레임워크) 명확

하지만 운영에서 곧바로 부딪히는 현실이 있습니다.

- 기능 1개 추가 = core + react + vue + 문서 + 검증 포인트 동시 증가
- 컴포넌트 하나 수정 = 양쪽 회귀 테스트/QA 동기화 필요
- 릴리즈 리드타임 증가
- 신규 기능 개발보다 “패리티 유지 작업” 비중이 커짐

즉, 구조 자체가 틀렸다기보다 **조직 규모 대비 운영 한계를 넘어서기 쉬운 구조**였던 겁니다.

## 중요한 전환: “동일 지원”에서 “레이어별 지원”으로

사례 팀이 선택한 전환은 매우 실용적입니다.

- **React**: 컴포넌트 레이어의 운영 축으로 집중
- **Vue**: 토큰/스타일/유틸리티(파운데이션) 중심 소비

이 결정은 기술적 후퇴가 아니라 운영 전략의 고도화입니다.

얻는 것:

- 컴포넌트 변경 흐름 단순화
- 릴리즈 예측 가능성 향상
- 기능 반영 속도 개선

포기하는 것:

- Vue 컴포넌트 레벨의 즉시 확장성 일부

유지하는 것:

- 시각 규칙/토큰 정합성
- 브랜드·테마 일관성

요약하면, “둘 다 완벽하게”가 아니라 **“어디를 완벽하게 할지 선택”**한 것입니다.

## 프론트엔드 팀이 바로 적용할 수 있는 의사결정 프레임

멀티 프레임워크 디자인 시스템을 운영 중이라면, 아래 순서로 판단하면 시행착오를 줄일 수 있습니다.

### 1) 레이어를 분리해서 목표를 정의한다

- 파운데이션 레이어(토큰/타이포/컬러/아이콘/유틸)
- 컴포넌트 레이어(Button, Modal, Input 등)
- 문서/플랫폼 레이어(스토리북, 가이드, QA 체계)

레이어마다 “동일 지원”이 필요한지 따로 판단해야 합니다.

### 2) 패리티(Parity)보다 리드타임(Lead Time)을 지표로 본다

다음 지표가 나빠지면 경고 신호입니다.

- 컴포넌트 1개 변경 시 평균 반영 시간
- 프레임워크 간 회귀 이슈 비율
- 릴리즈 지연의 원인 중 패리티 유지 비중

패리티는 목표가 아니라 결과물입니다.
리드타임이 무너지면 패리티도 결국 유지되지 않습니다.

### 3) “공통화할 것”보다 “의도적으로 분리할 것”을 먼저 합의한다

실무에서 가장 효과적인 질문은 이겁니다.

- “이 레이어는 반드시 두 프레임워크 동일해야 하는가?”
- “아니라면 어느 레이어까지만 공통으로 가져갈 것인가?”

초기 합의만 잘해도, 나중에 생기는 “왜 Vue가 React와 다르냐” 논쟁을 크게 줄일 수 있습니다.

### 4) 파운데이션 DX를 높여 팀 생산성을 회수한다

사례처럼 아래 영역은 공통화 투자 효율이 큽니다.

- Typography/Color를 CSS 변수 기반으로 노출
- Semantic Color 체계로 브랜드 변경 비용 분리
- Icon의 `currentColor` 정규화
- Shadow/Radius/Spacing 토큰화

컴포넌트 레벨 공통화 일부를 내려놓아도,
파운데이션 DX를 끌어올리면 제품 일관성과 개발 생산성을 동시에 챙길 수 있습니다.

## 결론: 디자인 시스템은 코드베이스가 아니라 ‘운영 시스템’이다

디자인 시스템의 실패는 보통 API 설계가 나빠서가 아니라,
팀이 감당 가능한 운영 범위를 넘어서는 순간 시작됩니다.

따라서 멀티 프레임워크 전략의 핵심은 다음 한 문장으로 정리됩니다.

> “동일하게 제공할 수 있는 능력”보다
> “지속적으로 운영할 수 있는 범위”를 먼저 설계하자.

장기적으로 살아남는 디자인 시스템은
가장 화려한 구조가 아니라 **가장 오래 유지 가능한 운영 모델** 위에서 만들어집니다.
