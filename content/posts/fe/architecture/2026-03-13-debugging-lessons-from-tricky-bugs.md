# 까다로운 버그에서 배운 9년의 디버깅 교훈

최근 KOFE Article의 최신 포스트인 **「까다로운 버그에서 배운 9년간의 교훈」**을 읽고, 프론트엔드 팀이 바로 적용할 수 있는 실천 항목 중심으로 재정리했다.

핵심 메시지는 단순하다.

> 버그는 “사람이 실수해서”가 아니라, **시스템이 실수를 허용하는 구조**에서 반복된다.

즉, 디버깅 역량은 개인의 감이 아니라 **코드 구조 + 테스트 전략 + 런타임 관측성**의 합으로 만들어야 한다.

---

## 1) 코딩 단계: 버그가 생기기 어려운 구조를 먼저 만든다

### 1-1. 상태 전이(transition)를 코드로 명시하기
프론트엔드 버그의 상당수는 `if` 분기가 늘어나면서 상태가 암묵적으로 꼬일 때 발생한다.

예: `idle → loading → success|error` 흐름이 컴포넌트 여기저기에 흩어져 있으면,
- 로딩 스피너가 영구 고정되거나
- 에러 상태에서 재시도 버튼이 비활성화되거나
- stale 데이터가 화면에 남는 문제가 나온다.

**실전 원칙**
- 상태를 문자열 상수/유니온으로 고정한다.
- 상태 전이 가능 경로를 reducer 또는 state machine으로 제한한다.
- “절대 나오면 안 되는 상태 조합”을 타입 단계에서 막는다.

```ts
type FetchState =
  | { status: 'idle' }
  | { status: 'loading' }
  | { status: 'success'; data: Portfolio[] }
  | { status: 'error'; reason: string }
```

이렇게 하면 `success`가 아닌 상태에서 `data`를 참조하는 실수를 컴파일 단계에서 차단할 수 있다.

### 1-2. 부수효과를 이벤트 핸들러 경계로 밀어내기
디버깅 난이도를 키우는 대표 원인은 “언제 API가 호출되는지 모르는 코드”다.

- 렌더 도중 파생 호출
- 의존성 배열이 복잡한 `useEffect`
- 여러 훅에서 같은 리소스를 중복 fetch

**실전 원칙**
- 부수효과는 이벤트(클릭/submit/route-change) 경계로 모은다.
- 네트워크 호출은 전용 계층(query hook/service)으로 격리한다.
- 컴포넌트는 “무엇을 보여줄지”에 집중하고, “어떻게 가져올지”는 위임한다.

### 1-3. 실패를 정상 흐름으로 설계하기
실무에서 장애를 만드는 건 성공 케이스가 아니라 실패 케이스다.

- 타임아웃
- 부분 성공(partial success)
- 레이트 리밋(429)
- 네트워크 단절

**실전 원칙**
- 각 API별 실패 정책(재시도/중단/백오프/사용자 메시지)을 문서화한다.
- 에러 객체를 표준화하여 UI에서 일관되게 처리한다.
- “재시도 가능/불가능”을 사용자에게 분명히 보여준다.

---

## 2) 테스트 단계: “재현 가능성”을 자동화한다

### 2-1. 단위 테스트는 규칙 검증, E2E는 사용자 시나리오 검증
테스트 부재가 문제인 것처럼 보이지만, 실제론 **테스트 역할이 뒤섞인 것**이 더 큰 문제다.

- 단위 테스트: 상태 전이, 파서, formatter, validation 등 결정적 로직
- 통합 테스트: API 응답 변화에 따른 화면 반응
- E2E: 실제 사용 플로우(로그인→조회→수정→저장)

한 레이어에서 모든 걸 잡으려 하면, 느리고 불안정해지고 유지보수성이 떨어진다.

### 2-2. 회귀 버그는 “버그 재현 테스트”를 먼저 작성
핫픽스 이후 동일 버그가 재발하는 이유는 간단하다. **재발 방지 안전장치가 없기 때문**이다.

**실전 체크리스트**
1. 버그를 최소 입력으로 재현하는 테스트 작성
2. 테스트가 실패하는 것 확인
3. 코드 수정
4. 테스트가 통과하는 것 확인
5. 같은 클래스의 인접 케이스 1~2개 추가

이 과정을 습관화하면 디버깅 경험이 팀 자산으로 축적된다.

### 2-3. Mock은 “단순화”가 아니라 “통제”를 목표로
Mock이 현실과 멀어질수록 테스트 신뢰도는 떨어진다.

- 응답 지연(latency)
- 순서 역전(out-of-order response)
- 500/429/timeout

이런 현상을 테스트에서 일부러 주입해야 실제 장애 대응력이 올라간다.

---

## 3) 런타임 관측성: 로컬에서 보이지 않는 버그를 보이게 만든다

### 3-1. 프론트엔드 로그는 “행동 단위”로 남긴다
console 로그를 무작정 늘리는 방식은 운영에서 거의 쓸모가 없다.

권장 이벤트 스키마 예시:
- `event`: `portfolio.save.clicked`
- `requestId`: 서버/클라이언트 상관키
- `userActionId`: 사용자 액션 단위 상관키
- `durationMs`
- `result`: success/failure
- `errorCode`

이 구조가 있으면 “누가/어떤 맥락에서/어느 구간에서 실패했는지”를 짧은 시간 안에 좁힐 수 있다.

### 3-2. 에러 리포팅은 stack trace + 도메인 맥락이 함께 있어야 한다
stack trace만 있으면 “어디서 터졌는지”는 보이지만 “왜 터졌는지”가 안 보인다.

그래서 다음 맥락을 같이 남겨야 한다.
- 현재 라우트
- 사용자 권한/플랜(민감정보 제외)
- API 응답 상태
- feature flag 상태
- 직전 사용자 액션

### 3-3. 성능 이슈도 버그다
UI가 느려서 사용자가 중복 클릭하면, 결국 중복 요청/중복 결제/중복 저장 같은 기능 버그로 확장된다.

따라서 Web Vitals/Interaction latency를 **품질 지표**로 함께 관리해야 한다.

---

## 4) 팀 운영 관점: 디버깅을 개인기에서 시스템으로 전환하기

### 디버깅 회고 템플릿(간단 버전)
- **증상:** 사용자에게 어떻게 보였는가?
- **근본 원인:** 코드/설계/운영 중 어디였는가?
- **탐지 실패 이유:** 왜 더 빨리 못 잡았는가?
- **재발 방지:** 테스트, 알림, 코드 구조 중 무엇을 바꿨는가?
- **확산 범위:** 유사 영역 점검 완료 여부

이 템플릿을 이슈 종료 조건에 포함하면, 디버깅 품질이 급격히 안정된다.

---

## 마무리

이번 글의 교훈을 프론트엔드 문맥으로 요약하면 다음 한 문장으로 정리된다.

> 좋은 디버깅은 “잘 고치는 능력”이 아니라, **같은 문제를 다시 만나지 않게 설계하는 능력**이다.

코딩 단계에서 상태와 효과를 명시하고,
테스트 단계에서 재현 가능성을 자동화하고,
운영 단계에서 관측성을 확보하면,
“까다로운 버그”는 운 나쁜 사건이 아니라 관리 가능한 리스크로 바뀐다.
