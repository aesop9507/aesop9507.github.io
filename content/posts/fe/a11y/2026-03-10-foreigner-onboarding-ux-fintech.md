## 왜 이 주제가 중요한가

금융 서비스의 성장 지표에서 **가입 전환율**은 기능 추가보다 훨씬 큰 임팩트를 낼 때가 많습니다.
특히 외국인 사용자에게는 “UI가 예쁘다/빠르다”보다 먼저, **첫 인증 단계를 통과할 수 있느냐**가 핵심입니다.

토스의 외국인 유저 리서치 사례는 이 문제를 매우 선명하게 보여줍니다.
핵심은 단순합니다.

- 이름 입력 포맷 불일치로 인증 반복 실패
- 실패 원인을 이해할 수 없는 에러 메시지
- 한국어 중심 주소 입력 흐름으로 검색 실패
- 결국 앱을 포기하고 오프라인 창구로 이동

즉, 기능 부족이 아니라 **입력 규칙과 피드백 설계 부족**이 이탈을 만듭니다.

---

## 리서치에서 드러난 본질: "사용자에게 데이터 정합성 책임을 넘겼다"

온보딩을 설계할 때 자주 하는 실수는,
백엔드 정합성 규칙(은행/통신사/신분증 데이터 일치)을 프론트에서 그대로 요구하는 것입니다.

예를 들어 사용자 입장에서는 다음 질문에 답할 방법이 없습니다.

- "내 이름을 띄어쓰기 없이 써야 하나?"
- "성-이름 순서를 바꿔야 하나?"
- "영문 대소문자/하이픈/공백 중 무엇이 기준인가?"

이 상황에서 “인증에 실패했습니다”만 반복하면,
사용자는 문제를 해결하는 것이 아니라 **무작위 실험**을 하게 됩니다.
이것이 UX 관점에서 가장 비싼 실패입니다.

---

## 프론트엔드 관점의 개선 원칙 4가지

## 1) 입력값이 아니라 **의도**를 받는다

사용자에게는 “정답 문자열”을 요구하지 말고,
프론트가 다중 포맷 후보를 생성해 백엔드 검증 API와 협력해야 합니다.

예: 이름 `BRAD PITT`

- `BRADPITT`
- `BRAD PITT`
- `PITT BRAD`
- 여권/외국인등록증 표기 규칙 기반 변형

핵심은 사용자가 포맷 규칙을 외워서 맞추는 구조를 없애는 것입니다.

---

## 2) 에러는 "거절"이 아니라 **다음 행동 안내**여야 한다

나쁜 에러:

- "인증 실패"

좋은 에러:

- "통신사 등록 이름과 일치하지 않아요. 공백을 제거한 영문 이름으로 다시 시도해 주세요."
- "5회 실패 전, 직원 연결 또는 문서 가이드를 바로 제공"

에러 메시지는 기술 로그가 아니라 **문제 해결 UI**입니다.

---

## 3) 주소 입력은 한글 타이핑 전제를 버린다

외국인 사용자 플로우에서는 다음이 기본이어야 합니다.

- 우편번호/영문주소/지번 동시 검색
- 결과 리스트에 행정구역 + 도로명 + 영문 병기
- 최근 입력 재시도 힌트("방금 입력한 우편번호 기준으로 다시 찾기")

즉, 검색엔진처럼 설계하고 폼처럼 보이게 해야 합니다.

---

## 4) 실패 누적 제한(예: 5회)을 UX 정책과 함께 설계한다

보안 정책으로 시도 횟수 제한은 필요합니다.
하지만 **제한만 있고 복구 경로가 없으면** 이탈만 증가합니다.

반드시 함께 제공해야 할 것:

- 남은 시도 횟수 가시화
- 실패 원인 카테고리 표시(이름/통신사/문서)
- 즉시 대체 경로(상담 연결, 서류 인증 전환, 예약 인증)

---

## 구현 체크리스트 (실무용)

### 입력 계층

- [ ] 이름 입력 컴포넌트에 다국어/다중 포맷 힌트 제공
- [ ] normalize pipeline(공백/하이픈/대소문자) 프론트 공통 모듈화
- [ ] 서버 검증 전 후보 포맷 자동 생성 옵션 제공

### 피드백 계층

- [ ] 에러 코드를 사용자 행동 단위 문구로 매핑
- [ ] 실패 시 다음 행동 CTA 1개 이상 강제
- [ ] 실패 횟수/잠금 상태를 시각적으로 상시 노출

### 검색 계층

- [ ] 주소 검색 입력 하나로 다중 키워드(우편번호/영문/지번) 처리
- [ ] 검색 결과 정렬 기준: 정확도 + 언어 친화성
- [ ] 찾지 못했을 때 fallback 시나리오(센터 찾기/상담 연결)

### 분석 계층

- [ ] 단계별 이탈 이벤트(입력 시작/검증 실패/재시도/완료) 로깅
- [ ] "실패 원인별 재시도 성공률" 대시보드화
- [ ] 국적/언어 세그먼트별 전환율 추적

## 정리

외국인 온보딩 문제는 "특수 사용자 케이스"가 아닙니다.
글로벌 사용자 확장을 준비하는 모든 서비스에서 먼저 터지는 **기본 UX 품질 문제**입니다.

좋은 프론트엔드 설계는 사용자의 실수를 줄이는 것을 넘어,
사용자가 **실패하더라도 다시 성공할 수 있게** 만듭니다.

금융 앱에서 이는 친절함의 문제가 아니라,
전환율과 신뢰를 동시에 지키는 제품 경쟁력입니다.
