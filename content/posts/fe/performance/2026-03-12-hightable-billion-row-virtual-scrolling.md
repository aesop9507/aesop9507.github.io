데이터 테이블은 보통 "행이 많아지면 느려진다" 수준에서 끝나지 않습니다.  
수백만~수십억 행 규모로 가면 **DOM 개수 최적화**보다 먼저, 브라우저 자체의 물리적 한계(최대 요소 높이, 스크롤바 픽셀 정밀도)와 맞닥뜨리게 됩니다.

이번 글은 Korean FE Article에서 소개된 HighTable 사례(원문 번역 링크 포함)를 바탕으로, 대규모 테이블 스크롤링을 실서비스 UI에서 어떻게 풀 수 있는지 프론트엔드 관점에서 재구성한 내용입니다.

---

## 왜 "일반적인 가상 스크롤"만으로는 부족한가

우리가 익숙한 가상 스크롤은 보통 다음 문제를 해결합니다.

- 보이는 영역만 렌더링해 DOM 수를 줄인다.
- 스크롤 위치에 맞춰 필요한 데이터만 가져온다.

하지만 행 수가 극단적으로 커지면 아래 문제가 추가됩니다.

1. **요소 최대 높이 한계**  
   브라우저는 무한 높이 요소를 허용하지 않습니다. 특정 임계값을 넘으면 캔버스/컨테이너 높이를 더 키울 수 없습니다.

2. **스크롤바 정밀도 한계**  
   스크롤바는 본질적으로 픽셀 단위로 동작합니다. 전체 높이를 강하게 압축하면, 1px 이동이 수천~수만 행 점프로 변합니다.

3. **접근성과 네이티브 동작 유지의 어려움**  
   커스텀 스크롤바/가짜 캔버스 접근은 구현은 쉬워 보이지만 키보드/보조기기 호환성에서 비용이 커집니다.

HighTable의 핵심은 이 세 문제를 **네이티브 HTML 스크롤을 유지하면서** 단계적으로 해결한 점입니다.

## HighTable의 5가지 기법 요약

### 1) 지연 로딩 (Lazy Loading)

초기 로딩에서 전체 데이터셋을 메모리에 올리지 않고, 현재 보이는 구간 중심으로 fetch 합니다.

- `getCell()`은 동기적으로 캐시 조회
- `fetch(range)`는 비동기로 누락 구간 로딩 후 resolve 이벤트 발행
- 스크롤 이벤트 때 필요한 range만 요청

**핵심 효과:**
- 메모리 사용량을 "전체 데이터 크기"가 아니라 "화면+버퍼 구간"에 비례하도록 제한

---

### 2) 테이블 슬라이스 (Table Slice)

DOM에는 전체 행을 그리지 않고, 보이는 행(+버퍼)만 `<tr>`로 유지합니다.

구조는 다음과 같습니다.

- `viewport`(스크롤 컨테이너)
- `canvas`(전체 높이를 대표하는 공간)
- `table slice`(실제 렌더되는 소수 행)

즉, 스크롤 위치는 큰 공간에서 유지하되 DOM은 항상 작게 유지합니다.

**핵심 효과:**
- 렌더링 비용 상한을 고정
- 행 수 증가와 무관하게 UI 반응성 유지

---

### 3) 무한 픽셀 (Infinite Pixels)

전체 이론 높이가 브라우저 허용 높이를 넘으면, `downscaleFactor`를 적용해 스크롤 공간을 압축합니다.

- 임계값 이하: 1px 스크롤 = 1px 콘텐츠 이동
- 임계값 이상: 1px 스크롤 = Npx 콘텐츠 이동

이를 통해 물리적 높이 제한을 넘지 않으면서도, 논리적으로는 수십억 행 전역 탐색이 가능합니다.

**장점:** 전역 이동 가능  
**단점:** 정밀 스크롤이 깨짐(행 건너뛰기 발생)

---

### 4) 픽셀 단위 정밀 스크롤 (Local/Global Hybrid)

3번의 단점을 보완하기 위해 스크롤을 두 모드로 분리합니다.

- **Global Scroll:** 스크롤바 드래그처럼 큰 점프
- **Local Scroll:** 휠/트랙패드의 미세 이동

내부 상태를 분리해 운영합니다.

- `globalAnchor`: 전역 기준점
- `localOffset`: 로컬 미세 이동 누적
- `scrollTop`: 직전 위치

작은 delta는 `localOffset`으로 처리해 연속 스크롤 감각을 유지하고, 큰 delta만 전역 점프로 반영합니다.

**핵심 효과:**
- 대규모 전역 탐색 + 근거리 정밀 탐색 동시 확보

---

### 5) 2단계 랜덤 접근 (Two-step Random Access)

"N번째 행으로 이동" 같은 요청은 한 번에 완벽 정렬하지 않고,

1) 전역 앵커를 목적지 인근으로 점프
2) 로컬 오프셋으로 정밀 보정

의 2단계로 수렴시킵니다.

이 접근은 브라우저 스크롤 정밀도 제약 안에서 안정적으로 특정 행에 도달하게 해줍니다.

## 실무 적용 관점: 우리가 바로 가져갈 설계 원칙

### A. 스크롤을 "UI 이벤트"가 아니라 "좌표계 변환"으로 보라

대규모 데이터 그리드에서는 스크롤 처리 자체가 핵심 도메인 로직입니다.

- 물리 좌표(브라우저 scrollTop)
- 논리 좌표(행 index)
- 압축 좌표(downscaled position)

이 3개를 분리하지 않으면, 행 점프/정렬 오차/포커스 깨짐 버그가 반복됩니다.

### B. 데이터 계층과 렌더링 계층을 명확히 분리하라

HighTable의 DataFrame 인터페이스처럼,

- 데이터 로딩/캐시(비동기)
- 셀 읽기(동기)

를 분리하면 렌더 사이클이 단순해지고, 소스가 파일/HTTP/로컬메모리로 바뀌어도 UI는 유지됩니다.

### C. 접근성은 "나중에"가 아니라 구조 단계에서 결정된다

헤더를 별도 오버레이가 아니라 `<table><thead>` 내부 sticky로 유지하는 선택은 스크린리더와 키보드 탐색에서 큰 차이를 만듭니다.

"보이는 UI"보다 "의미론적 구조"를 먼저 지키는 편이 장기 유지보수에 유리합니다.

## Next.js/React 프로젝트에서의 구현 체크리스트

대규모 데이터 테이블을 직접 구현한다면 최소 아래를 권장합니다.

1. **고정 rowHeight 전제부터 시작** (가변 높이는 2차 단계)
2. **overscan 버퍼**를 상하로 별도 운영
3. **scroll event -> rAF 배치 처리**로 리렌더 폭주 방지
4. **fetch 중복 제거 + 범위 병합(range coalescing)**
5. **로딩 실패 셀 fallback UI**(blank cell 방치 금지)
6. **키보드 탐색/포커스 복원 전략** 문서화
7. **랜덤 접근 API (`scrollToRow`)**는 2단계 정렬 방식 채택

## 마무리

수십억 행 스케일에서 프론트엔드 성능 문제는 단순 최적화가 아니라 **브라우저 제약을 수학적으로 우회하는 설계 문제**에 가깝습니다.

HighTable 사례가 주는 가장 큰 인사이트는 다음 한 줄로 정리됩니다.

> "렌더링 최적화"만으로는 부족하다.  
> "스크롤 의미 체계"를 설계해야 대규모 데이터 UI가 완성된다.

데이터 제품(로그 탐색기, BI 그리드, 대용량 백오피스 테이블)을 다루는 팀이라면, 이 5단계 구조를 그대로 복제하기보다 **좌표계 분리 + 하이브리드 스크롤 상태 관리**라는 원칙부터 도입해 보시길 추천합니다.
