# OpenStack 기반 하이브리드 클라우드 아키텍처 - 토스 사례 분석

퍼블릭 클라우드만으로는 비용과 데이터 주권 문제를 해결하기 어려운 대규모 핀테크 서비스에서, 하이브리드 클라우드는 점점 현실적인 선택지가 되고 있습니다. 토스가 OpenStack 기반 프라이빗 클라우드를 직접 구축하고 퍼블릭 클라우드와 Active-Active로 운영하는 사례를 분석합니다.

**출처:** [Toss Tech Blog](https://toss.tech)

---

## 왜 하이브리드 클라우드인가

### 퍼블릭 클라우드의 한계

대규모 핀테크 서비스에서 퍼블릭 클라우드만 사용할 때 발생하는 문제들이 있습니다:

- **비용 증가**: 트래픽 규모가 커질수록 퍼블릭 클라우드 비용이 기하급수적으로 증가
- **벤더 종속(Vendor Lock-in)**: 특정 클라우드 프로바이더에 종속되면 마이그레이션 비용이 급등
- **데이터 주권**: 금융 데이터의 물리적 위치와 접근 제어에 대한 규제 요건

### 토스의 전략적 선택

토스는 이러한 한계를 극복하기 위해 **OpenStack 기반 프라이빗 클라우드를 직접 구축**하는 결정을 내렸습니다. 단순히 비용 절감이 아닌, 워크로드 특성에 맞는 최적의 인프라 배치를 목표로 한 전략적 접근입니다.

## Active-Active 하이브리드 구성

### 기존 DR 방식과의 차이

전통적인 하이브리드 구성은 퍼블릭을 Primary, 프라이빗을 DR(Disaster Recovery)로 사용하는 Active-Passive 패턴이 일반적입니다. 토스는 여기서 한 단계 더 나아가 **Active-Active** 구성을 채택했습니다.

```
┌─────────────────────┐     ┌─────────────────────┐
│   퍼블릭 클라우드      │ ←→ │  프라이빗 클라우드      │
│   (Active)          │     │  (Active)            │
│                     │     │  OpenStack 기반       │
│  - 탄력적 워크로드    │     │  - 안정적 워크로드     │
│  - 버스트 트래픽     │     │  - 데이터 주권 요건    │
│  - 실험적 서비스     │     │  - 비용 최적화 대상    │
└─────────────────────┘     └─────────────────────┘
```

### Active-Active의 이점

| 항목 | Active-Passive | Active-Active |
|------|---------------|---------------|
| 리소스 활용률 | Primary만 활용 | 양쪽 모두 활용 |
| 장애 대응 | 페일오버 시간 필요 | 즉시 트래픽 분산 |
| 비용 효율 | DR 리소스 유휴 | 유휴 리소스 최소화 |
| 운영 복잡도 | 상대적 낮음 | 높음 (동기화 필요) |

Active-Active 구성은 운영 복잡도가 높지만, 리소스 효율과 고가용성을 동시에 확보할 수 있습니다.

## OpenStack 기반 프라이빗 클라우드 구축

### OpenStack을 선택한 이유

- **오픈소스**: 벤더 종속 없이 인프라 전체를 제어 가능
- **성숙한 생태계**: 컴퓨트(Nova), 네트워크(Neutron), 스토리지(Cinder) 등 검증된 컴포넌트
- **커스터마이제이션**: 금융 서비스 특성에 맞는 네트워크 정책, 보안 설정 적용 가능

### 핵심 구축 요소

1. **IaC(Infrastructure as Code)**: 인프라 규모가 커질수록 수동 관리는 불가능. Terraform/Ansible 기반의 자동화가 필수
2. **통합 모니터링**: 퍼블릭과 프라이빗 양쪽의 상태를 단일 대시보드에서 관찰
3. **네트워크 설계**: 두 환경 간의 안정적인 연결과 낮은 레이턴시 보장

## 실무 적용 시 고려사항

### 워크로드 분류 기준

하이브리드 클라우드의 핵심은 **워크로드 특성에 맞는 배치 전략**입니다:

| 워크로드 특성 | 적합한 환경 | 이유 |
|-------------|-----------|------|
| 트래픽 변동이 큰 서비스 | 퍼블릭 | 탄력적 스케일링 활용 |
| 안정적/예측 가능한 서비스 | 프라이빗 | 비용 최적화 |
| 데이터 주권 요건 | 프라이빗 | 물리적 위치 제어 |
| 실험적/신규 서비스 | 퍼블릭 | 빠른 프로비저닝 |
| GPU/ML 워크로드 | 상황에 따라 | 비용 vs 가용성 비교 |

### 운영 자동화의 중요성

Active-Active 하이브리드 환경에서는 자동화 없이는 운영이 불가능합니다:

- **배포 자동화**: 양쪽 환경에 일관된 배포 파이프라인
- **장애 감지/복구**: 자동 헬스체크와 트래픽 라우팅 조정
- **구성 관리**: 환경 간 설정 드리프트 방지

## 결론

토스의 사례는 "클라우드 전환"이 단순히 퍼블릭 클라우드로의 마이그레이션이 아님을 보여줍니다. **워크로드 특성을 분석하고, 각 환경의 장점을 극대화하는 하이브리드 전략**이 대규모 서비스에서는 더 현실적인 접근입니다.

특히 금융 서비스처럼 데이터 주권과 비용 최적화가 동시에 요구되는 도메인에서, OpenStack 기반 프라이빗 클라우드 + 퍼블릭 Active-Active 구성은 검토할 가치가 충분합니다.

### 핵심 요약

| 포인트 | 내용 |
|--------|------|
| **전략** | 워크로드 특성 기반 인프라 배치 |
| **구성** | Active-Active 하이브리드 (유휴 리소스 최소화) |
| **기반** | OpenStack (벤더 종속 탈피) |
| **필수 요소** | IaC, 통합 모니터링, 자동화된 운영 |
