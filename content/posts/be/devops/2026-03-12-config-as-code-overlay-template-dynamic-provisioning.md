대규모 백엔드 운영에서 장애는 코드보다 **설정**에서 먼저 시작되는 경우가 많습니다.

- 같은 의미의 설정이 서비스마다 복붙되며 드리프트가 생기고
- 긴 문자열 환경 변수(JVM 옵션 등)에서 오타가 숨어들고
- 배치 인프라는 리소스 경합 때문에 간헐 장애를 만든다

토스 기술 블로그의 「수천 개의 API/BATCH 서버를 하나의 설정 체계로 관리하기」를 읽고, 백엔드 팀이 바로 적용할 수 있는 형태로 핵심을 재구성했습니다.

---

## 1) 문제의 본질: 설정 중복은 운영 리스크를 선형이 아니라 **기하급수**로 키운다

서버 수가 늘어날수록 코드보다 설정 변경이 더 자주 일어납니다.

- 신규 환경 변수 추가
- 메모리/GC 튜닝
- 특정 클러스터별 프로파일 분기
- 배치 스케줄/실행 파라미터 조정

이때 설정이 서비스 단위로 복사돼 있으면, 작은 변경에도 N개 파일을 동시에 수정해야 하고 일부 누락/오타가 곧바로 장애로 연결됩니다.

핵심 포인트는 단순합니다.

> 설정을 “문서”가 아니라 “실행 가능한 소프트웨어”로 관리해야 한다.

---

## 2) API 서버 설정: Overlay 아키텍처로 책임을 계층화한다

토스 사례의 첫 번째 축은 **오버레이(Overlay)** 입니다. 설정을 하나의 거대한 YAML이 아니라, 책임별 계층으로 분리합니다.

예시 계층:

- global: 전역 기본값
- phase: dev/staging/live 환경 기본값
- cluster: 클러스터별 차이
- app/service: 서비스별 차이

하위 계층이 상위 계층을 덮어쓰는 우선순위를 명확히 두면, 변경 영향 범위를 예측할 수 있습니다.

### 실무 적용 체크리스트

1. 공통값은 최상위(global)로 끌어올린다.
2. 예외값만 하위 계층에서 오버라이드한다.
3. 파일명 컨벤션으로 계층 의미를 강제한다. (`values-phase-live-common.yaml` 등)
4. CI/CD가 이 계층 구조를 그대로 해석하도록 통합한다.

이 구조가 만들어지면 “개발 환경만 카나리 생략” 같은 배포 정책도 코드 수정 없이 설정 계층에서 제어할 수 있습니다.

---

## 3) Overlay만으로는 부족하다: Template 패턴으로 긴 문자열 드리프트를 막는다

실무에서 가장 까다로운 건 배열/긴 문자열 기반 설정입니다. JVM 옵션처럼 일부 값만 바꾸고 싶어도 전체 문자열을 다시 써야 하면 복붙이 재발합니다.

해법은 **템플릿화**입니다.

- `JVM_OPTION`을 템플릿으로 선언
- `MAX_HEAP`, `REGION_SIZE` 같은 변수만 계층별로 override
- 최종 렌더링은 파이프라인에서 수행

이 패턴의 장점:

- 변경 단위를 “문자열 전체”에서 “변수 1개”로 축소
- diff 가독성 향상
- 잘못된 재정의(예: 오타 포함 전체 문자열 재입력) 확률 감소

즉, Overlay가 “어디서 바꿀지”를 해결한다면, Template는 “어떻게 안전하게 바꿀지”를 해결합니다.

## 4) Batch 운영: 선언형 Job + 동적 프로비저닝으로 인프라 변동성을 흡수한다

배치는 API보다 더 위험합니다. 실행 실패가 즉시 사용자 장애로 보이지 않더라도, 정산/집계 누락 같은 치명적 데이터 오류로 이어질 수 있기 때문입니다.

토스 사례에서 인상적인 부분은 Jenkins를 버리지 않고도 운영 품질을 끌어올린 점입니다.

### 4-1. Job-DSL 어댑터(선언형 인터페이스)

개발자가 Jenkins 내부 구조를 몰라도 다음만 선언하면 되게 만듭니다.

- 어떤 애플리케이션을
- 어떤 JDK/JVM 옵션으로
- 어떤 스케줄과 모니터링 채널로 실행할지

그리고 DevOps는 공통 기능(APM/프로파일러/덤프/모니터링)을 중앙에서 한 번만 표준화합니다.

이 구조의 본질은 **플랫폼 팀이 복잡성을 흡수**하고, 제품 팀은 **의도를 선언**하게 만드는 것입니다.

### 4-2. Dynamic Provisioning

고정 워커에서 여러 배치가 메모리를 경합하면 OOM/비정상 종료가 반복됩니다. 이를 해결하려면 “한 노드에 몇 개 프로세스” 최적화 게임보다, 워크로드 단위로 리소스를 동적으로 붙였다 떼는 구조가 낫습니다.

- 배치 실행 시 노드 동적 할당
- 유휴 시 자동 회수
- 리소스 격리로 메모리 경합 감소
- 일회성 노드 전략으로 OS 패치/보안 적용 단순화

결국 안정성과 비용을 함께 잡을 수 있습니다.

## 5) 백엔드 팀을 위한 도입 순서 (작게 시작해서 확장하기)

아래 순서로 가면 리스크를 낮출 수 있습니다.

1. **설정 인벤토리 작성**
   - 전역/환경/클러스터/서비스 값 분류
2. **Overlay 1차 도입**
   - 기존 복붙 YAML을 계층으로 재배치
3. **고위험 문자열 템플릿화**
   - JVM 옵션, 커넥션 옵션, 스케줄 표현식부터
4. **설정 테스트 추가**
   - 운영 규칙 위반(STAGING에 LIVE 프로파일 등) 자동 검증
5. **배치 선언형 인터페이스 구축**
   - Job 템플릿/빌더 제공
6. **동적 프로비저닝 단계적 적용**
   - 고메모리/고충돌 배치부터 우선 전환

---

## 6) 핵심 교훈: “설정도 제품”이라는 관점이 운영 성숙도를 만든다

이번 사례의 가장 중요한 메시지는 기술 선택 자체가 아닙니다. Kubernetes냐 Jenkins냐보다 더 중요한 건 다음입니다.

- 설정 변경에도 설계 원칙이 있는가?
- 설정 변경이 테스트 가능한가?
- 설정 인프라가 요구사항 변화에 진화 가능한가?

설정을 코드처럼 다루기 시작하면, DevOps는 단순 운영자가 아니라 **조직의 변경 비용을 낮추는 플랫폼 엔지니어링 팀**이 됩니다. 그리고 백엔드 팀은 인프라 잡음에서 벗어나 도메인 문제 해결에 집중할 수 있습니다.

---

## 정리

- Overlay: 책임/우선순위 분리
- Template: 긴 문자열 설정의 안전한 부분 변경
- 선언형 Job 인터페이스: 개발자 경험 표준화
- Dynamic Provisioning: 배치 리소스 경합 제거 + 비용 최적화

대규모 백엔드에서 “설정”은 부수 작업이 아닙니다. 설정 체계가 곧 서비스 신뢰성과 개발 속도를 결정합니다.
