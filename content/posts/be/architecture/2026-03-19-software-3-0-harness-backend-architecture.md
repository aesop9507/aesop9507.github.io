## 한 줄 요약

LLM의 성능만으로는 서비스가 되지 않습니다. **백엔드가 해오던 아키텍처 원칙(레이어 분리, 책임 경계, 실패 복구)**을 Harness로 재구성해야 에이전트가 운영 가능한 시스템이 됩니다.

---

## 왜 지금 백엔드가 이 주제를 잡아야 하나

Software 3.0 담론의 핵심은 “프롬프트가 프로그램이 된다”는 주장입니다. 하지만 프로덕션에서는 이렇게 끝나지 않습니다.

- 코드베이스 접근 권한은 어떻게 제어할 것인가?
- 외부 API 호출 실패는 누가 재시도할 것인가?
- 잘못된 판단으로 위험 작업을 실행하려 할 때 누가 멈출 것인가?

즉, 문제는 모델의 지능보다 **시스템 책임 배치**입니다. 이 지점은 전형적인 백엔드 문제입니다.

---

## 원문 핵심: LLM을 일하게 만드는 것은 Harness다

Toss Tech의 「Embracing the Software 3.0 Era」가 특히 좋았던 이유는, LLM 한계를 막연히 말하지 않고 **운영 계층으로 번역**했다는 점입니다.

- 컨텍스트 한계 → 메모리/세션 관리 필요
- 환각 리스크 → grounding/RAG 필요
- 외부 시스템 미접근 → 툴/MCP 필요
- 상태 관리 부재 → 오케스트레이션 필요

이건 사실 백엔드에서 오래 다뤄온 주제와 같습니다.

- stateless 서비스의 세션 설계
- 외부 의존성 장애 격리
- 데이터 일관성/재시도/보상 트랜잭션
- 인터페이스와 어댑터 분리

결론: **Agent Runtime은 새로운 이름의 분산시스템**입니다.

---

## 백엔드 관점으로 재해석한 Harness 레이어

원문의 비유(슬래시 커맨드=컨트롤러, 서브에이전트=서비스, MCP=어댑터)는 실무 적용성이 높습니다. 다만 운영 관점에서 아래처럼 더 명확히 나누는 것이 좋습니다.

### 1) Intent Layer (Controller)

역할:
- 사용자 의도를 정규화
- 권한·정책 검증
- 실행 계획 선택

백엔드 체크포인트:
- 입력 스키마 강제(JSON Schema/Zod)
- 권한 모델(읽기/쓰기/삭제) 분리
- idempotency key 적용(중복 실행 방지)

### 2) Orchestration Layer (Service)

역할:
- 작업 분할/순서 제어
- 실패 시 재시도/중단/보상
- 비용·지연시간 예산 관리

백엔드 체크포인트:
- Step 단위 timeout, retry budget
- saga-like 보상 플로우
- circuit breaker(외부 툴 과부하 방지)

### 3) Capability Layer (Domain Skills)

역할:
- 단일 책임 도구(코드리뷰, 테스트 생성, 문서 갱신 등)
- 재사용 가능한 기능 단위

백엔드 체크포인트:
- SRP 위반 감지(“God Skill” 방지)
- 입출력 계약 테스트(contract test)
- 버전 관리(v1/v2 병행)

### 4) Integration Layer (MCP / Adapter)

역할:
- DB, GitHub, CI, Slack 등 외부 시스템 연결
- 도메인 로직과 외부 프로토콜 분리

백엔드 체크포인트:
- adapter timeout/재시도 정책 일원화
- 에러 매핑(외부 429/5xx → 내부 표준 에러)
- 감사 로그(누가, 언제, 무엇을 호출했는지)

## HITL은 “예외 처리 전략”으로 설계해야 한다

원문에서 가장 실무적인 차별점은 HITL(Human-in-the-Loop)을 **에이전트의 강점**으로 본 부분입니다.

기존 백엔드:
- 불확실하면 exception throw
- 혹은 디폴트 정책으로 진행

에이전트 백엔드:
- 불확실하면 질문으로 승격
- 승인 후 계속 진행

이때 중요한 건 “언제 질문할지”를 정책화하는 것입니다.

### 질문이 필요한 작업

- 되돌리기 어려운 작업(삭제, 배포, 결제/정산 변경)
- 보안/컴플라이언스 영향 작업
- 다수의 합리적 선택지가 있는 작업

### 자동 진행 가능한 작업

- 재실행 안전(read-only, 문서 정리, 테스트 실행)
- 실패 시 자동 롤백 가능한 작업
- 조직 컨벤션으로 정답이 정해진 작업

핵심: HITL은 친절 기능이 아니라 **리스크 관리 장치**입니다.

## Tokens Are the New RAM: 운영 지표를 반드시 둬야 한다

원문 후반부의 “토큰은 새로운 RAM”이라는 문장은 백엔드 운영팀에게 특히 중요합니다.

권장 지표:
- 요청당 총 토큰 사용량
- 단계별 컨텍스트 크기
- 모델 응답 실패율(타임아웃/중단)
- 재시도 횟수와 비용 증가율

운영 원칙:
1. **정적 규칙은 스크립트로 이동** (LLM이 매번 해석하지 않게)
2. **대규모 작업은 단계 분할** (한 번에 전체 분석 금지)
3. **컨텍스트 TTL 운영** (오래된 히스토리 자동 축약)

토큰 최적화는 비용 절감 문제가 아니라 **성능과 신뢰성 문제**입니다.

## 안티패턴: 기존 백엔드 냄새가 그대로 반복된다

원문의 안티패턴 매핑은 실제로 매우 정확합니다.

- God Class → God Skill
- Spaghetti Code → Spaghetti Prompt/CLAUDE.md
- Tight Coupling → 특정 툴/응답 포맷 하드코딩
- Circular Dependency → skill 간 순환 호출

실전에서 추가로 자주 보이는 문제:

- **Observability 부재:** 왜 실패했는지 추적 불가
- **권한 과다:** 읽기 작업인데 write 권한까지 부여
- **평가 부재:** 정확도만 보고 재현성/안정성 미측정

결국 백엔드에서 익숙한 운영 원칙(최소권한, 로그, SLO, 회귀테스트)이 그대로 필요합니다.

---

## 바로 적용 가능한 백엔드 실행 체크리스트

### A. 계약(Contract)
- [ ] 각 skill/MCP의 입력·출력 스키마 문서화
- [ ] 실패 코드 표준화(VALIDATION_ERROR, RATE_LIMIT, TIMEOUT 등)

### B. 안전(Safety)
- [ ] 위험 작업은 HITL 승인 게이트 필수
- [ ] 승인 로그와 실행 로그를 같은 trace-id로 연결

### C. 운영(Operations)
- [ ] 단계별 latency/비용 대시보드 구축
- [ ] 429/5xx 급증 시 자동 degraded mode 진입

### D. 품질(Quality)
- [ ] golden task 기반 회귀평가 세트 유지
- [ ] 모델 교체 시 동일 task A/B 비교 후 점진 롤아웃

## 마무리

Software 3.0은 “코드가 끝났다”는 선언이 아니라, **백엔드의 책임 범위가 확장됐다**는 신호에 가깝습니다.

이제 백엔드 엔지니어는 API 서버만 안정화하는 역할을 넘어,

- 에이전트의 실행 경계
- 도구 접근 권한
- 실패 복구 정책
- 인간 승인 흐름

까지 설계해야 합니다.

모델은 점점 좋아지겠지만, 프로덕션의 승패는 여전히 같습니다.

> **좋은 아키텍처는 성능이 아니라, 실패했을 때도 시스템을 설명 가능하게 만든다.**
