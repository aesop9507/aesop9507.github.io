# 결제 백엔드 보안 아키텍처 실전: 경계 보안에서 제로트러스트로 가는 4단계 전환

토스 테크의 **「경계 보안부터 제로트러스트 보안까지, 고도화 여정」**은 보안 조직 관점의 사례를 잘 보여줍니다.  
이번 글에서는 이를 **백엔드/플랫폼 아키텍처 관점**으로 재해석해, 실제 서비스 운영팀이 적용 가능한 형태로 정리합니다.

핵심은 한 문장입니다.

> 단일 방어선(Perimeter-only)에서 살아남는 시대는 끝났다.  
> 이제는 `경계 + 내부 + 런타임 + 접근제어`를 하나의 운영체계로 묶어야 한다.

---

## 1) 문제 정의: 왜 "경계 보안만"으로는 실패하는가

결제·정산·금융 API는 다음 조건 때문에 공격 표면이 넓습니다.

- 외부 연동(가맹점/파트너/벤더) 다수
- 하이브리드 인프라(IDC + Cloud)
- 고가용성 요구로 인해 빠른 배포·복구 필요
- 트래픽 대부분이 TLS 암호화

여기서 경계 보안만 강조하면 다음 문제가 발생합니다.

1. **암호화 트래픽 블라인드 스팟**: TLS 내부 페이로드 가시성 부족  
2. **내부 침투 이후 측면 이동(Lateral Movement)** 방어 취약  
3. **컨테이너 동적 워크로드**를 전통 서버 도구가 따라가지 못함  
4. **업무망 접근 모델(VPN/IP 기반)**이 최소권한·지속검증을 보장하지 못함

즉, 공격자는 “한 번 뚫으면 내부에서 오래 머무르는” 쪽으로 유리해집니다.

---

## 2) 전환 전략: 4개의 방어축을 순차 도입

토스페이먼츠 사례를 백엔드 팀 실행 모델로 바꾸면 아래 4단계입니다.

### 단계 A. 경계 보안 고도화 (Encrypted Traffic Visibility 확보)

목표: **보이지 않던 HTTPS 위협을 보이게 만들기**

- TLS 복호화 기반 탐지 체계(적법/정책 범위 내)
- IDC와 AWS 간 보안정책 동등성 확보
- 파트너/가맹점 연계 트래픽에 대한 피드백 루프 구축
  - 단순 차단이 아니라 “원인 + 조치 가이드” 제공

이 단계의 성패는 차단율보다 **가시성 지표**로 봐야 합니다.

- 복호화 가시성 커버리지
- 탐지 룰 적중률/오탐률
- 파트너 재발률 감소

### 단계 B. 내부망 보안 (Assume Breach 모델)

목표: **경계가 뚫려도 확산을 제한**

- IDC: Wazuh 계열의 통합 로그/무결성/행위 모니터링
- AWS: GuardDuty 기반 권한상승·비정상 런타임 이벤트 감지
- 이상행위 이벤트의 공통 분류 체계(Severity, Tactic, Asset Criticality)

실무 포인트는 “탐지 그 자체”가 아니라 **조치 자동화율**입니다.

- 수동 티켓 생성 → 반자동 격리 → 자동 격리로 단계 상향
- 탐지 후 MTTR보다 `MTTContain`(확산 억제 시간) 측정

### 단계 C. 컨테이너 런타임 보안 (Final Runtime Guard)

목표: **실행 시점 위협을 잡는다**

정적 스캔(SCA/SAST)만으로는 런타임 행위를 막기 어렵습니다.

- IDC K8s: Falco 기반 syscall 룰 탐지
  - 예: `/etc/shadow` 접근, 신규 바이너리 실행
- Cloud K8s/EKS: GuardDuty 컨테이너 런타임 이벤트 연계
- 이벤트를 SIEM/알림/자동 조치로 연결

중요한 건 룰 개수가 아니라 **워크로드 맥락**입니다.

- 네임스페이스/서비스/데이터 민감도 기반 차등 대응
- 동일 이벤트라도 결제 코어와 배치 워커의 임계값을 다르게

### 단계 D. 제로트러스트 업무망 (Identity + Device + Context)

목표: **“사내망이면 신뢰” 모델 폐기**

- IP/VPN 중심에서 사용자·단말·상태 기반 동적 접근제어로 전환
- ZTNA + EDR 연계로 접속 전/중 지속 검증
- 조직/직무 변경 시 권한 자동 재계산

결국 핵심은 네트워크가 아니라 **신원(Identity)과 상태(Posture)**입니다.

---

## 3) 백엔드 관점에서 가장 중요한 설계 원칙 3가지

### 원칙 1. 보안은 “도구 도입”이 아니라 “데이터 파이프라인”이다

WAF, EDR, GuardDuty, Falco를 따로 운영하면 알람은 늘고 대응은 느려집니다.

- 이벤트 표준 스키마
- 상관분석 규칙
- 대응 Playbook 자동화

까지 포함해야 “운영 가능한 보안”이 됩니다.

### 원칙 2. 탐지 정확도보다 먼저 대응 경로를 단순화하라

온콜이 새벽 3시에 따라갈 수 없는 절차는 실패합니다.

- Critical 이벤트는 3단계 이내로 격리 가능해야 함
- 조치 실패 시 롤백 가능한 안전장치 필요
- 변경 이력과 감사 로그를 기본값으로 남겨야 함

### 원칙 3. 최소권한은 정책 문서가 아니라 상태 머신이다

권한은 정적 ACL 파일이 아니라 “계속 변하는 상태”입니다.

- 사람/서비스 계정 수명주기 관리
- 단말 보안 상태 연동
- 리소스 민감도 태깅

이 3개가 묶여야 제로트러스트가 작동합니다.

---

## 4) 도입 로드맵 (12주 예시)

### 1~4주: 가시성 확보
- TLS/경계 이벤트 수집 정비
- 자산 인벤토리(서비스/데이터/권한) 최신화
- 가장 위험한 5개 경로 우선 식별

### 5~8주: 내부/런타임 방어 강화
- IDC/AWS 이상행위 탐지 룰 표준화
- 컨테이너 런타임 핵심 룰(민감 파일 접근/신규 바이너리/권한상승) 우선 배치
- 고위험 이벤트 자동 격리 PoC

### 9~12주: 접근제어 전환
- VPN/IP 예외정책 축소
- 사용자·단말 기반 동적 접근정책 적용
- 권한 변경 자동화 + 감사 대시보드 오픈

---

## 5) 트레이드오프: 실무에서 반드시 합의해야 할 것

이 전환은 비용이 있습니다.

- 초기엔 오탐 증가
- 운영팀 학습 부담 증가
- 정책 충돌로 배포 속도 일시 저하

하지만 얻는 것은 큽니다.

- 침해 후 확산 반경 축소
- 탐지→조치 리드타임 단축
- 감사/규제 대응 신뢰도 향상

특히 결제 도메인에서는 “조금 느리더라도 안전한 시스템”이 장기적으로 더 빠릅니다. 대형 사고 한 번의 비용이, 예방 체계 구축 비용을 대부분 상회하기 때문입니다.

---

## 마무리

경계 보안에서 제로트러스트로의 전환은 제품 하나를 붙이는 프로젝트가 아닙니다.  
백엔드 아키텍처, IAM, 런타임 관측, 온콜 운영을 **하나의 시스템**으로 재설계하는 일입니다.

이번 사례가 주는 실무 메시지는 명확합니다.

- "막는다"보다 "보이게 한다"
- "탐지"보다 "즉시 조치한다"
- "신뢰한다"보다 "계속 검증한다"

이 세 가지 원칙으로 접근하면, 하이브리드 환경에서도 보안과 개발 속도를 함께 지킬 수 있습니다.
