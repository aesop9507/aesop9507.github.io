반복적인 백엔드 업무 중 하나는 **코드를 작성하기 전의 태스크 정제**입니다.

- 기획 문서를 읽고
- 구현 단위로 분해하고
- 담당자/라벨/우선순위를 붙여
- Jira 티켓으로 등록하는 과정

이 과정은 중요하지만, 팀이 커질수록 소모되는 시간이 빠르게 증가합니다.
GC Company 기술블로그의 「기획서에서부터 Jira 태스크 업로드까지, Tasky」는 이 반복 구간을 **AI + 워크플로우 자동화**로 줄인 사례입니다.

이 글에서는 해당 사례를 백엔드 관점으로 재구성해, 실제 팀에서 도입 가능한 형태로 정리합니다.

---

## 1) 왜 이 문제가 백엔드 아키텍처 이슈인가

많은 팀이 이 문제를 “PM 도구 자동화” 정도로 보지만, 실상은 **생산 시스템의 입력 품질 문제**입니다.

입력(요구사항)이 불안정하면 결과(코드/릴리즈)도 흔들립니다.

### 입력 품질이 낮을 때 생기는 전형적 장애

1. 같은 기능인데 팀원마다 다른 해석으로 구현
2. 선행/의존 관계 누락으로 스프린트 중간 블로킹
3. 운영/모니터링/롤백 태스크가 빠져 배포 후 장애 확률 증가
4. 티켓 품질 편차로 리뷰/QA 부하 증가

즉, Tasky 같은 시스템은 편의 기능이 아니라
**백엔드 전달체계의 표준화 계층**입니다.

---

## 2) Tasky를 백엔드 파이프라인으로 보면: 4단계 변환기

기획서→티켓 자동화는 본질적으로 “문서 변환”이 아니라 **의도 변환**입니다.

### Stage A. 문서 수집(Ingestion)

- 입력: Confluence 페이지/하위 섹션
- 처리: 섹션 단위 파싱, 표/체크리스트/코드블록 정규화
- 출력: 구조화된 중간 표현(요구사항 단위 JSON)

핵심은 원문을 바로 LLM에 던지지 않고,
먼저 **파싱 가능한 단위**로 잘게 나누는 것입니다.

### Stage B. 태스크 분해(Decomposition)

- 기능 요구를 구현 가능한 작업 단위로 쪼갬
- API/DB/FE/QA/운영 태스크를 분리
- 선행관계(`depends_on`)를 명시

이 단계에서 “잘게 쪼개되 지나치게 잘게 쪼개지 않는” 균형이 중요합니다.

### Stage C. 정책 검증(Validation)

자동 생성 결과를 바로 Jira로 보내면 위험합니다.
검증 규칙이 필수입니다.

- 모든 태스크에 `definition_of_done` 존재 여부
- 운영성 항목(로그/알람/모니터링) 포함 여부
- 보안/권한 관련 요구 누락 여부
- 예상 포인트 범위 벗어난 이상치 탐지

### Stage D. Jira 동기화(Publish)

- 프로젝트 키/이슈 타입/컴포넌트/라벨 매핑
- 에픽-스토리-서브태스크 계층 구성
- 멱등성 키(idempotency key)로 중복 생성 방지

여기까지가 갖춰져야 “자동 생성”이 아니라 “자동 운영”이 됩니다.

---

## 3) 아키텍처 설계 원칙: LLM보다 중요한 것은 계약(Contract)

Tasky류 시스템에서 실패 포인트는 모델 정확도보다
**입출력 계약 부재**에서 먼저 발생합니다.

### 권장 계약 예시

```json
{
  "task_title": "string",
  "task_type": "api|db|infra|qa|docs",
  "owner_role": "backend|frontend|devops|pmo",
  "acceptance_criteria": ["string"],
  "depends_on": ["task_id"],
  "risk_level": "low|medium|high",
  "observability": {
    "logging": true,
    "metrics": true,
    "alerting": false
  }
}
```

LLM 출력은 항상 흔들릴 수 있으므로,
최종 저장 전에는 스키마 검증(Zod/Pydantic/JSON Schema)을 반드시 거쳐야 합니다.

---

## 4) 운영 관점에서 꼭 넣어야 할 세 가지

### 4-1. Human-in-the-loop 승인 게이트

자동 생성된 티켓은 “초안”으로 취급하고,
리드가 승인한 뒤에만 실제 Jira 생성을 허용합니다.

- 초안 단계: 수정 가능
- 승인 단계: 감사 로그 기록
- 반려 단계: 재생성/수정 후 재검토

### 4-2. 변경 추적(Traceability)

모든 Jira 티켓은 원문 기획의 섹션 ID를 추적 가능해야 합니다.

- `source_doc_id`
- `source_section_id`
- `generation_version`

이 추적성이 있어야 변경 요청 시 안전하게 동기화할 수 있습니다.

### 4-3. 재동기화 전략

기획은 계속 바뀝니다. “최초 1회 생성”으로 끝나지 않습니다.

- 문서 변경 감지(diff)
- 영향 범위 계산(어떤 티켓이 영향 받는지)
- 자동 코멘트 or 자동 상태 전환

핵심은 생성보다 **동기화**입니다.

---

## 5) 도입 순서: 작은 팀도 가능한 최소 경로

1. Confluence 1개 스페이스만 대상으로 시작
2. 이슈 타입 2~3개(Story/Task/Bug)만 우선 자동화
3. 스키마 검증 + 승인 게이트부터 구축
4. 그 다음에 의존관계/포인트 추정 자동화 확장

초기에는 정확도보다 **재현성 있는 출력**을 먼저 확보해야 합니다.

---

## 6) 이번 사례의 실전 교훈

GC Company 사례에서 배울 수 있는 핵심은 다음입니다.

- 자동화 대상은 “복잡한 알고리즘”보다 “반복되는 고비용 협업 구간”이어야 한다.
- AI 도입의 품질은 프롬프트보다 **운영 정책(검증/승인/추적)**이 좌우한다.
- 업무 속도를 높이려면 생성 속도보다 **입력-출력 표준화**가 먼저다.

백엔드 팀 입장에서는 Tasky 같은 워크플로우를 도입할 때,
“티켓을 자동으로 만든다”가 아니라
“요구사항이 실행 계획으로 변환되는 계약을 만든다”는 관점이 더 중요합니다.

---

## 마무리

기획서를 Jira로 옮기는 일은 사소해 보이지만,
실제로는 팀의 실행 품질을 결정하는 첫 관문입니다.

Tasky 사례가 보여준 방향은 명확합니다.

- 문서 파싱
- 구조화 분해
- 정책 검증
- 승인 후 발행

이 4단계를 파이프라인으로 고정하면,
개발자는 반복 입력 작업에서 벗어나 **문제 해결 그 자체**에 더 많은 시간을 쓸 수 있습니다.
