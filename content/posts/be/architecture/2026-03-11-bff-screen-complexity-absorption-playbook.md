## 왜 지금 BFF를 다시 봐야 하나

MSA를 도입한 조직에서 자주 만나는 역설이 있습니다.

- 도메인은 잘 분리됐다.
- 그런데 화면은 더 복잡해졌다.

도메인 팀이 각자의 Bounded Context 안에서 API를 잘 제공해도, 실제 사용자 경험은 “화면 단위”로 발생합니다. 화면 하나를 그리기 위해 검색/주문/혜택/결제/리뷰 API를 각각 호출하고, iOS·Android·Web가 동일한 조합 로직을 개별 구현하면 클라이언트는 점점 무거워집니다.

이번 글은 GC Company(여기어때) 기술블로그의 **“화면 단위 복잡성을 흡수하다: 여기어때 BFF의 기록”**을 바탕으로, BFF를 단순 중계 계층이 아닌 **복잡성 흡수 계층**으로 운영하는 방법을 정리한 글입니다.

---

## 핵심 요약: BFF는 “API 개수” 문제가 아니라 “책임 배치” 문제다

여기어때 사례의 핵심은 다음 한 문장으로 요약됩니다.

> 도메인은 분리하되, 화면 조합 책임은 BFF에 모은다.

많은 팀이 BFF를 “프론트 편의용 API” 정도로 이해하지만, 실전에서 중요한 건 **어떤 책임을 어디에 둘지**입니다.

### BFF가 가져와야 할 책임

1. **Aggregate 책임**
   - 여러 도메인 API를 호출해 화면에 필요한 단일 응답으로 통합
2. **ViewModel 변환 책임**
   - 도메인 모델을 화면 친화적인 형태로 변환
3. **상태 분기 책임**
   - 결제 상태/동의 상태/노출 정책 같은 UI 분기 로직을 중앙화
4. **회복 탄력성 책임**
   - 재시도, 타임아웃, Fast Fail, Fallback 정책을 서버 계층에서 일관 적용

클라이언트는 “화면 렌더링”에 집중하고, BFF는 “화면 구성에 필요한 데이터 책임”을 맡는 구조가 됩니다.

---

## 여기어때 사례에서 배울 수 있는 설계 포인트

### 1) API 통합은 단순 병합이 아니라 ‘화면 계약(contract)’ 설계다

광고센터 주문 상세 예시처럼, 실제 화면은 7~8개 API의 조합을 요구합니다. 문제는 호출 수만이 아닙니다.

- 결제 수단이 자동이체인지
- 계약 동의 상태가 어떤지
- 주문 상태가 취소/진행/완료인지

이 상태 조합에 따라 응답 구조와 CTA 노출 조건이 달라집니다.

즉 BFF는 “데이터를 붙이는 곳”이 아니라,
**화면 계약을 안정적으로 제공하는 계층**입니다.

실무 팁:

- BFF 응답을 화면 단위 DTO로 명시적으로 고정
- 분기 로직을 `if` 난립이 아니라 정책 객체/규칙 엔진으로 분리
- “클라이언트 조건식 금지” 원칙을 문서화

### 2) MSA + BFF에서 가장 위험한 건 장애 전파(Cascading Failure)

여기어때 팀이 강조한 것처럼, BFF는 다수 하위 API에 의존하기 때문에 지연 하나가 전체를 잠식할 수 있습니다.

권장 순서:

1. **짧은 타임아웃**을 기본값으로 둔다.
2. **재시도는 제한적**으로만 허용한다. (지수 백오프 + 최대 횟수 제한)
3. 재시도 실패 시 **즉시 Fast Fail/Fallback** 한다.
4. 실패율·지연 임계치 초과 시 **Circuit Breaker**로 호출 자체를 차단한다.

중요한 점은 “성공률 100%”가 아니라 **시스템 생존성**입니다. BFF가 죽지 않아야 나머지 정상 경로가 살아남습니다.

### 3) BFF의 성능 병목은 CPU보다 메모리에서 먼저 온다

조합 계층은 데이터를 모으는 특성상 힙 압박이 큽니다.

- 대량 리스트를 메모리에서 join/sort
- 병렬 처리 남용
- 객체 생성 폭증

이 패턴은 GC 증가 → 지연 증가 → 스레드 정체로 이어집니다.

여기어때 사례의 대응(페이징 강제 + Caffeine 캐시)은 매우 현실적입니다.

실무 체크리스트:

- 하위 API는 기본적으로 **pagination required**
- BFF에서 “전체 조회 후 필터링” 금지
- cache hit/miss, eviction, heap usage 대시보드 상시 모니터링
- 스트림 병렬화는 프로파일링 후 제한적으로 도입

## BFF 도입 전/후 아키텍처를 어떻게 판단할까

### BFF 도입이 유효한 신호

- 동일한 조합 로직이 여러 클라이언트에 반복됨
- 화면 변경 때 앱 3플랫폼이 동시에 깨짐
- 도메인 API 수는 늘어나는데 프론트 릴리즈 속도는 떨어짐
- 화면 단위 분기 로직이 프론트 코드에 산재함

### BFF 도입을 미루는 게 나은 신호

- 단일 클라이언트 + 단순 CRUD 중심
- 화면별 조합 요구가 거의 없음
- 팀이 운영 복잡도(모니터링/장애대응)를 감당할 준비가 안 됨

BFF는 항상 정답이 아니라 **복잡성 이동 전략**입니다. 클라이언트 복잡성을 서버로 옮기는 대신, 서버 운영 성숙도가 필요합니다.

---

## 운영 가능한 BFF를 위한 최소 규칙 7가지

1. **Endpoint는 화면/유즈케이스 기준으로 설계**한다. (`/order-detail`처럼)
2. **응답 스키마 버전 관리**를 한다. (UI 계약 깨짐 방지)
3. **하위 호출별 timeout/retry/circuit-breaker**를 분리 설정한다.
4. **부분 실패 정책**을 명문화한다. (무조건 실패 vs 일부 degrade)
5. **페이징과 캐시 전략**을 도메인 팀과 계약 수준으로 맞춘다.
6. **관측성(Tracing/Metric/Log Correlation)** 없이는 운영하지 않는다.
7. **BFF는 도메인 비즈니스 소유자가 아니다.** 화면 조합 책임까지만 가진다.

---

## 마무리: 좋은 BFF는 “클라이언트를 가볍게”만들고 “서버를 예측 가능하게” 만든다

여기어때 사례가 좋은 이유는 BFF를 이상론으로 설명하지 않고, 실제 운영에서 맞닥뜨린 장애 전파·메모리 부담까지 드러냈기 때문입니다.

정리하면,

- MSA는 도메인을 잘게 나누고,
- BFF는 사용자 화면에서 다시 의미 있게 묶습니다.

결국 중요한 건 계층 추가 자체가 아니라,
**어떤 복잡성을 어디에서 감당할지에 대한 팀의 합의**입니다.

BFF를 도입하려는 팀이라면 “API를 하나로 줄이자”보다 먼저,
“화면 단위 책임을 중앙화하고 운영 리스크를 통제할 준비가 됐는가?”를 점검해야 합니다.

그 질문에 “예”라고 답할 수 있다면,
BFF는 프론트 생산성을 높이는 도구를 넘어 **시스템 진화 속도를 지키는 아키텍처 안전장치**가 됩니다.
