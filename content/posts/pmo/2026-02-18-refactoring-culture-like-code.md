## 개요

프로젝트가 진행되면 될수록 조직은 자연스럽게 굳어갑니다. "원래 이렇게 해왔어요"라는 말이 익숙함의 이유만으로 비효율을 정당화하기 시작하고, 루틴은 형식만 남은 의무가 됩니다.

우아한형제들 기술블로그에서 최근 "우리는 코드처럼 문화도 리팩토링한다"라는 흥미로운 글이 게시되었습니다. 커머스웹프론트팀(커머프팀)이 팀의 비효율을 개선하기 위해 조직 구조, 소통 방식, 기록 체계를 어떻게 지속적으로 리팩토링했는지를 다루는 내용입니다.

이 글에서는 PMO(Product/Project Management Office) 관점에서, 커머프팀의 조직 리팩토링 여정에서 얻을 수 있는 실전 인사이트를 정리하고 실무에 적용할 수 있는 구체적인 프레임워크를 제안합니다.

## 조직 리팩토링의 본질: 왜 필요한가

### 레거시 코드처럼 굳어버린 조직

소프트웨어 개발에서 "레거시 코드"는 수정하기 어렵고 이해하기 힘든 코드를 말합니다. 조직도 마찬가지입니다:

```yaml
레거시 조직의 증상:
  구조적 경직성:
    - 담당자가 없으면 업무가 멈춤
    - 파트 간 사일로(Silo) 심화
    - 도메인 경계를 넘나드는 협업 어려움
  소통 비용 증가:
    - 20명이 한자리에 모인 회의 시간: 2시간 이상
    - 불필요한 context switching 잦음
    - 슬랙 스레드가 휘발성으로 사라짐
  기록의 파편화:
    - 개인이 가진 지식이 공유되지 않음
    - 기술적 결정의 맥락이 유실됨
    - 온보딩 비용 증가
```

### 리팩토링의 목표

조직 리팩토링은 단순한 구조 변경이 아닙니다:

```
조직 리팩토링의 목표:
  ┌─────────────────────────────────────────┐
  │  유연한 기동력: 변화에 빠르게 대응      │
  │  투명한 협업: 맥락이 공유되는 환경    │
  │  지속적 개선: "원래 이렇게"에 도전    │
  └─────────────────────────────────────────┘
```

## 1단계: 조직 구조 리팩토링 - 경계 없는 파트

### 전통적인 분할 방식의 한계

커머프팀은 거의 20명 규모의 FE 개발자로 구성된 거대 팀입니다. 초기에는 다음과 같은 전통적인 분할 방식을 검토했습니다:

| 분할 방식 | 한계 | 이유 |
|---------|------|------|
| 프로젝트별 | 리소스 불균형 | 과제 발생 시기 불규칙으로 특정 파트에 과업 몰림 |
| 페이지/퍼널 단위 | 전체 맥락 상실 | 맡은 지면 외의 영역 이해도 낮아짐 |
| 서비스 vs 백오피스 | 시너지 차단 | 상호 교류 막혀 통합 어려움 |

공통 원인: **구성원을 특정 업무 도메인에 지나치게 강하게 묶어둠**

### 경계 없는 파트(Boundary-less Parts)

커머프팀이 선택한 해결책은 **"경계 없는 파트"**입니다:

```
경계 없는 파트의 핵심:
  ├─ 도메인 고정 해제: 특정 서비스/페이지에 얽매이지 않음
  ├─ 유연한 배분: 업무 요청에 따라 동적 리소스 배분
  └─ 상호 지원: 빈자리를 유동적으로 메워주는 협업
```

### R&R에서 R&E로

전통적인 **R&R(Role & Responsibility)**은 개인을 특정 역할에 가두는 경직된 구조입니다. 커머프팀은 이를 **R&E(Responsibility & Expandability)**로 확장했습니다:

```yaml
R&R (Role & Responsibility):
  내 일의 경계를 뚜렷하게 정의
  주어진 역할만 수행

R&E (Responsibility & Expandability):
  내 일의 경계를 두지 않고 문제를 끝까지 해결
  책임을 넘어서 경계를 허물고 업무 영역을 확장
  서로를 돕는 협업
```

이는 우아한형제들의 일하는 원칙 **"Own It"**과 정확히 맞닿아 있습니다.

### PMO 관점: R&E 도입 가이드

#### 1. R&E 정의 템플릿

```markdown
# 팀 R&E 문서

## 핵심 원칙
"내 일의 경계를 두지 않고 문제를 끝까지 해결한다"

## 팀 공통 책임 (Responsibility)
- [ ] 팀의 목표 달성을 최우선으로 생각
- [ ] 맥락을 공유하고 지식을 문서화
- [ ] 동료를 돕고 빈자리를 메운다

## 개별 확장 가능 영역 (Expandability)
- 개인의 강점과 관심사
- 추가적으로 기여하고 싶은 도메인
- 새롭게 학습하고 싶은 기술

## 협업 규칙
- 업무 요청 → 파트장 싱크업 → 3개 파트로 배분
- 빈자리 발생 → 유동적으로 지원
- 과제 완료 → 팀 전체에 맥락 공유
```

#### 2. R&E 운영 체크리스트

| 단계 | 확인 사항 | 주의점 |
|------|-----------|--------|
| **설정** | 개별 확장 가능 영역 정의 | 강요하지 않고 자발적 참여 유도 |
| **배분** | 업무를 유동적으로 할당 | 과제의 성격과 개인의 상황 고려 |
| **지원** | 빈자리를 신속히 메우기 | 특정 인원에 과도한 의존 방지 |
| **공유** | 완료 후 맥락 공유 | ADR, 위클리 등 활용 |
| **회고** | 정기적 검토와 조정 | 3개월마다 R&E 재정의 |

#### 3. 부작용 방지

```
R&E 도입 시 부작용과 완화책:

  부작용 1: 지식 파편화
  └─ 완화: ADR(아키텍처 결정 기록) 강화

  부작품 2: Context Switching 피로
  └─ 완화: 연속성 있는 후속 과제는 같은 팀원 이어감

  부작품 3: 파트 간 사일로화
  └─ 완화: 파트는 배분 단위일 뿐, 별도 규칙 금지
```

## 2단계: 소통 방식 리팩토링 - 비동기 협업

### 위클리의 문제: 2시간 반의 산소 부족

20명이 한자리에 모여 한마디씩 의견을 보태면 회의 시간은 자연스럽게 늘어납니다:

```yaml
전통적인 위클리 문제:
  시간: 2시간 반 이상
  초점: 개별 업무 보고
  결과: 산소 부족, 집중력 흐름

  해결책 없는 대안:
    └─ Slack 스레드 이어가기: 논의 깊이 부족, 휘발성
```

### 불판: 가장 뜨거울 수 있는 곳

커머프팀이 찾은 해결책은 **"불판"**입니다. GitLab Issue를 기술 논의 전용 공간으로 사용합니다:

```
불판의 장점:
  익숙함: 항상 코드 리뷰로 소통하던 GitLab Issue
  맥락: 담당자, 라벨, 마감일 등 다양한 컨텍스트
  지속성: 스레드처럼 밀려나지 않고 그 자리에 남음
  타이밍: 고민이 생긴 순간 올리고 자유롭게 참여
  비동기: 회의 일정 잡을 필요 없이 논의 가능
```

### 위클리의 변화

불판 도입 후 위클리는 다음과 같이 변화했습니다:

```yaml
변화 전: 2시간 30분
  └─ 개별 업무 보고, 긴 논의

변화 후: 1시간 20분
  └─ 불판 요약 + 꼬리에 꼬리를 무는 질문

프로세스:
  1. 불판에 고민 올림 → Slack 알림
  2. 2~3일간 비동기 논의
  3. 위클리에서 요약 공유
  4. 추가 논의 필요 시 다시 불판으로
```

### PMO 관점: 불판 도입 가이드

#### 1. 불판 설정 템플릿

```markdown
# 불판(Bulpan) 운영 가이드

## 목적
기술적 고민과 문제를 비동기적으로 논의하여 효율적인 협업 환경 구축

## 도구
- GitLab Issue / GitHub Issue / Jira Issue

## 라벨링
- 📋 종류: 기획/기술/버그/개선
- 🔥 긴급도: 높음/중간/낮음
- 📁 파트: Frontend/Backend/DevOps/Design
- 🎯 상태: 논의중/결정보류/완료

## 운영 프로세스
1. 고민이 생기면 불판에 이슈 생성
2. Slack 채널에 알림 자동 발송
3. 관심 있는 팀원 자유롭게 참여
4. 2~3일간 비동기 논의 진행
5. 위클리에서 요약 공유
6. 추가 논의는 다시 불판에서

## 규칙
- 책임감 있는 토론: 인신공격 금지, 논점 중심
- 투명한 의사결정: 결정 과정 명확히 기록
- 적시한 마감: 결론이 나지 않으면 상태 변경
```

#### 2. 불판 효과 측정

| 지표 | 도입 전 | 도입 후 | 개선 |
|------|--------|--------|------|
| 위클리 시간 | 150분 | 80분 | 47% 단축 |
| 참여율 | 60% | 85% | 42% 증가 |
| 문제 해결 시간 | 7일 | 3일 | 57% 단축 |

## 3단계: 루틴 리팩토링 - "원래 이렇게"에 도전

### 굳어버린 루틴의 징후

커머프팀은 다음과 같은 신호를 포착했습니다:

```yaml
루틴 리팩토링이 필요한 신호:
  데일리:
    └─ 제목만 말하고 끝남
    └─ "도움이 필요한 곳" 공유가 아닌 보고 중심
  스프린트:
    └─ 일정이 들쭉날쭉함
    └─ 실제 일정과 맞지 않음
  회고:
    └─ 늘 해오던 거니까 방치
```

### 루틴 리팩토링 질문: "이걸 왜 하고 있지?"

커머프팀은 목적을 다시 정의하고 남길 것과 버릴 것을 명확히 했습니다:

| 루틴 | 이전 방식 | 리팩토링 후 |
|------|----------|------------|
| **스프린트** | 추상적인 SP, 플래닝 포커 | 실제 투입 시간(1SP=4시간), 단순화 |
| **데일리** | 길고 불규칙한 보고 | 15분 고정, 3가지만 짧게 공유 |
| **회의** | 전원 참여 필수 | 필요한 사람만 별도 모임 |

### 스프린트 운영 단순화

```yaml
리팩토링 전:
  추상적인 난이도 점수(SP)
  함께 모여 난이도 산정 (플래닝 포커)
  규칙 많고 이해 어려움

리팩토링 후:
  실제 투입 시간 (1SP = 4시간)
  플래닝 포커 과감히 제거
  긴급 이슈도 스프린트 흐름 안에서 관리
  단순함이 효율성
```

### 데일리 재정의

```yaml
데일리의 새로운 정의:
  목적: 보고가 아니라 "도움이 필요하거나 막힌 지점 찾기"

  공유 항목:
    1. 어제 완료한 일 (간단히)
    2. 오늘 할 일 (간단히)
    3. 막히는 장애 요소 (중요!)

  운영:
    └─ 파트별 Slack 스레드로 통일
    └─ 15분 고정
    └─ 깊은 논의 필요 시 별도 모임
```

### PMO 관점: 루틴 정기 검토 프로세스

```
루틴 정기 검주 프로세스:
  월: 정기 설문 통해 팀원 피드백 수집
  분: PMO와 팀장이 피드백 분석
  주: 리팩토링 방안 논의 및 결정
  적용: 2주간 시도 후 효과 측정
  다시 반복: "원래 이렇게"에 도전하는 문화
```

## 4단계: 기록 체계 리팩토링 - 1Pager에서 ADR로

### 1Pager의 한계

커머프팀은 오랫동안 1Pager(원페이저)를 표준으로 사용했습니다. 하지만 시간이 지날수록 본래의 목적을 잃고 무거워졌습니다:

```yaml
1Pager의 한계:
  형식: 기획서 링크, 피그마 링크, API 문서, Slack 논의 섞임
  내용: 방대해지고 가독성 저하
  질문: "왜 이렇게 구현했지?" → 여러 문서 뒤져야 함
  문제: 결과 중심, 맥락 부족
```

### ADR(Architectural Decision Record) 도입

1Pager를 가볍게 만드는 시도를 거쳐, 결국 ADR이라는 새로운 방식을 도입했습니다:

```
ADR 구조:
  Context (맥락): 어떤 문제 상황에서 논의가 시작되었는가?
  Decision (결정): 여러 선택지 중 무엇을 선택했고, 그 이유는 무엇인가?
  Consequences (결과): 그 결정으로 얻는 이득과 감수해야 할 비용은 무엇인가?
  Retrospective (회고): 진행하며 얻은 인사이트나 교훈은 무엇인가?
```

### ADR의 장점

| 장점 | 설명 |
|------|------|
| **의사결정 독립성** | 하나의 결정이 하나의 문서로 독립 |
| **맥락 보존** | 문제의 배경과 고민의 흐름 온전히 담김 |
| **히스토리 추적** | 새로운 ADR을 차곡차곡 쌓는 방식 |
| **지식 문서화** | 개인이 가진 지식을 팀 차원의 자산으로 |

### PMO 관점: ADR 도입 가이드

#### 1. ADR 작성 체크리스트

| 항목 | 설명 |
|------|------|
| **Context** | 현재 시스템 상황, 문제점, 영향 범위, 제약 조건 |
| **Decision** | 선택된 옵션, 대안들과 비교, 선택 이유 |
| **Consequences** | 긍정적 결과(이득), 부정적 결과(비용) |
| **Retrospective** | 진행하며 얻은 인사이트, 교훈 |
| **References** | 관련 링크, 문서, 토론 |

#### 2. ADR 템플릿

```markdown
# ADR-XXX: [제목]
```

## 5단계: 자동화 - 귀찮음을 레거시로 남기지 않는다

### 자동화의 철학

커머프팀의 자동화 철학은 명확합니다:

> 자동화는 그저 편함을 쫓는 작업이 아닙니다. 반복적인 입력과 수작업을 줄여 사람이 해야 할 판단과 문제 해결에 집중할 수 있는 환경을 만들어내는 과정입니다.

### 대표적인 자동화 사례

#### 1. Jira 이슈 관리 자동화

```yaml
자동화 전:
  시작일, 종료일, 릴리즈 연동, Story Point 등 직접 입력
  불필요한 에너지 소모

자동화 후:
  티켓을 '완료' 상태로 변경 → 시스템이 종료일 자동 기록
  Story Point 기입 → 예상 일정 자동 계산
  릴리즈 날짜 지정 → 해당 날짜의 릴리즈 버전 자동 생성
  기한 지난 티켓 → 주기적으로 Slack DM 알림
```

#### 2. 장애 알림 자동화

```yaml
자동화 전:
  장애 발생 → "누가 이거 확인할 수 있을까요?" 물음
  허비되는 시간

자동화 후:
  Sentry 최적화 → 특정 레벨 이상 에러 발생 시
  → 팀 Slack 채널로 즉시 알림
  → 담당 파트 자동 멘션
  → 담당자가 즉시 문제 인지 및 해결
```

#### 3. 장애 선포 자동화

```yaml
자동화 전:
  장애 채널 생성, 유관 부서 담당자 초대
  긴박한 상황에서 절차 챙기는 것 어려움

자동화 후:
  팀 채널에서 /선포 또는 /장애선포 입력
  → 회사 표준 포맷에 맞는 장애 채널 자동 생성
  → 유관 부서 담당자 자동 초대
  → 골든타임 사수, 대응 속도와 정확도 개선
```

#### 4. 릴리즈 노트 자동화

```yaml
자동화 전:
  배포 전 점검 항목 수동 확인
  롤백 방안 수동 기억

자동화 후:
  Jira 티켓에 배포 날짜 지정
  → 릴리즈 플랜 문서 자동 생성
  → 배포 전 점검 항목 구조화 표시
  → 롤백 방안 직관적 파악
  → 배포 안정성 크게 향상
```

### PMO 관점: 자동화 체크리스트

```
자동화 타겟 선정 체크리스트:
  반복성: 일상적으로 반복되는 작업인가?
  수작업: 사람이 직접 입력해야 하는가?
  에러 가능성: 실수가 발생할 수 있는가?
  시간 소모: 충분한 시간이 들어가는가?
  명확한 규칙: 자동화할 수 있을 만큼 규칙이 명확한가?

  5개 중 3개 이상 예 → 자동화 후보
```

## 실전 적용: PMO가 조직 리팩토링을 주도하는 방법

### 1단계: 현상 진단

PMO는 먼저 조직의 비효율을 진단해야 합니다:

```yaml
진단 질문:
  구조:
    └─ 특정 인원이 부재하면 업무가 멈추는가?
    └─ 파트 간 사일로가 존재하는가?
  소통:
    └─ 회의 시간이 너무 긴가?
    └─ 중요한 논의가 휘발성으로 사라지는가?
  루틴:
    └─ "원래 이렇게"라는 말이 자주 나오는가?
    └─ 형식만 남은 의무가 있는가?
  기록:
    └─ 기술적 결정의 맥락이 유실되는가?
    └─ 새로운 팀원 온보딩이 어려운가?
```

### 2단계: 실험 설계

PMO는 리팩토링을 위한 실험을 설계합니다:

```
실험 설계 프레임워크:
  목적: 무엇을 개선하고 싶은가?
  가설: 어떤 시도를 해볼 것인가?
  기간: 얼마 동안 시도할 것인가?
  성공 지표: 무엇으로 성공을 측정할 것인가?
  리스크: 어떤 부작용이 있을 수 있는가?
  완화책: 부작용을 어떻게 줄일 것인가?
```

### 3단계: 점진적 도입

한 번에 모든 것을 바꾸지 않습니다:

```yaml
점진적 도입 전략:
  1개월: 불판 도입 (소통 리팩토링)
  2개월: ADR 도입 (기록 체계 리팩토링)
  3개월: 데일리 재정의 (루틴 리팩토링)
  4개월: R&E 도입 (조직 구조 리팩토링)
  지속: 자동화 확장

  각 단계마다:
    └─ 효과 측정
    └─ 팀 피드백 수집
    └─ 다음 단계 결정
```

### 4단계: 효과 측정과 회고

PMO는 지속적으로 효과를 측정하고 회고해야 합니다:

| 지표 | 측정 방법 | 목표 |
|------|----------|------|
| **회의 시간** | 위클리, 데일리 시간 추적 | 30% 단축 |
| **참여율** | 불판 참여자 수 | 80% 이상 |
| **문서화율** | ADR 작성 건수 | 중요 결정 100% |
| **온보딩 시간** | 새 팀원 생산성 달성 시간 | 50% 단축 |
| **버스 팩터** | 핵심 인원 부재 시 대응력 | 3명 이상 |

## 조직 문화의 핵심: 실패가 아니라 과정

### 심리적 안전감(Psychological Safety)

커머프팀은 다음을 중요하게 생각합니다:

> 팀이 심리적으로 안전한 울타리가 되어야 합니다. 반대 의견을 냈다고 눈치를 보지 않아도 되고, 실수를 했다고 위축되지 않아도 되며, 당당하게 모른다고 말할 수 있는 분위기가 기본이 되었으면 합니다.

### 도전적인 문화

동시에 안전함에 안주하지 않는 문화입니다:

```yaml
도전적인 문화의 특성:
  질문: "이 구조가 정말 최선일까?"
  시도: "새롭게 시도해볼 수는 없을까?"
  실패 허용: 실패가 아니라 과정으로 받아들임
  함께 성장: 의도적으로 함께 실험하는 구조
```

### 질문이 스터디가 되는 팀

커머프팀은 업무를 하다 자연스럽게 생긴 기술적 고민과 문제 의식을 팀 전체로 끌어올리고 함께 해결하는 구조를 만들기 위해 정기적으로 팀 스터디를 진행합니다:

```
실제 스터디 사례:
  인프라 구조와 비용 이해 부족
  └─ AWS 스터디 진행
  └─ 인프라 구조 살펴보고 더 나은 대안 검토
  └─ 비용 절감 방법 탐구
  └─ 결과: 인프라 관점 단단해짐, 불필요 비용 절감
  └─ 추가: 팀원 전원 AWS 자격증 취득
```

## 결론: 지속적인 조직 진화

커머프팀의 이야기는 완성된 성공담이 아닙니다. 시도하고 실패하고 다시 시도하는 지속적인 과정입니다.

PMO의 역할은 완벽한 해결책을 제시하는 것이 아니라, 다음과 같은 질문을 던질 수 있는 환경을 만드는 것입니다:

```
PMO가 던져야 할 질문:
  이 구조가 정말 최선일까?
  새롭게 시도해볼 수는 없을까?
  이 일을 굳이 사람이 계속해야 하나?
  지식이 개인에게 갇혀 있지는 않은가?
  기록이 미래의 우리에게 길을 안내하고 있는가?
```

### 조직 리팩토링의 마인드셋

```yaml
조직 리팩토링의 핵심 마인드셋:
  완벽함이 아닌 진보:
    └─ 작은 불편함부터 손보기 시작
  익숙함에 도전:
    └─ "원래 이렇게"라는 말이 들리면 의심하기
  지속적 개선:
    └─ 한 번 리팩토링했다고 끝이 아님
  함께 성장:
    └─ 혼자가 아닌 팀으로서의 여정
```

> 기록은 더 이상 의무적으로 작성하는 귀찮은 문서가 아니라, 미래의 우리에게 길을 안내하는 팀의 기억 장치가 되었습니다.
> — 우아한형제들 기술블로그

PMO로서, 여러분의 팀도 코드처럼 문화를 리팩토링하는 여정을 시작해 보시는 건 어떨까요? 작은 시도부터 시작하면, 그 시도들이 모여 더 단단하고 유연한 조직을 만들어 갈 것입니다.
