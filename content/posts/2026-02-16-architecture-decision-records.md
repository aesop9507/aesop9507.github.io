## 들어가며

최근 우아한형제들 기술블로그에서 "우리는 코드처럼 문화도 리팩토링한다"라는 흥미로운 글을 읽었습니다. 커머스웹프론트팀이 팀의 비효율을 개선하기 위해 조직 구조, 소통 방식, 기록 체계를 어떻게 리팩토링했는지를 다루는 내용이었는데, 특히 **ADR (Architecture Decision Record)** 부분이 Backend 개발자로서 깊은 공감을 불러일으켰습니다.

이 글에서는 ADR이 무엇이고, 왜 Backend 개발자에게 중요한지, 그리고 실무에 어떻게 적용할 수 있는지 정리해보고자 합니다.

## ADR이란?

ADR (Architecture Decision Record)은 "아키텍처 결정 기록"이라는 의미로, 시스템의 기술적 결정과 그 맥락을 체계적으로 문서화하는 방법입니다.

우아한형제들 블로그에서 정의한 ADR 구조는 다음과 같습니다:

- **Context (맥락)**: 어떤 문제 상황에서 논의가 시작되었는가?
- **Decision (결정)**: 여러 선택지 중 무엇을 선택했고, 그 이유는 무엇인가?
- **Consequences (결과)**: 그 결정으로 얻는 이득과 감수해야 할 비용은 무엇인가?
- **Retrospective (회고)**: 진행하며 얻은 인사이트나 교훈은 무엇인가?

## 왜 Backend 개발자에게 ADR이 중요한가?

### 1. "왜 이렇게 구현했지?"라는 질문에 답하기

Backend 시스템은 복잡합니다. 데이터베이스 스키마, API 설계, 캐싱 전략, 메시지 큐, 마이크로서비스 분리 등 수많은 기술적 결정이 필요합니다.

시간이 지나면 "왜 Postgres를 썼지?", "왜 이 API는 이렇게 설계되었지?", "왜 여기서 Redis를 썼지?"와 같은 질문이 자연스럽게 발생합니다. 이때 ADR이 없다면:

- 당시 결정했던 사람을 찾아야 함
- 기억이 흐려져서 정확한 맥락을 알 수 없음
- 새로운 사람이 onboarding될 때 어려움

### 2. 지식의 저장소를 "사람"에서 "문서"로

우아한형제들 블로그에서 언급한 중요한 통찰입니다.

> 지식의 파편화와 의존성을 막기 위해, 지식의 저장소를 "사람"에서 "문서"로 옮기는 작업에 집중했습니다.

Backend 시스템은 시간이 지나면 지날수록 복잡해집니다. 특정 인원이 모든 맥락을 기억하는 것은 불가능합니다. ADR을 통해 개인이 파편적으로 가지고 있던 지식을 팀 차원의 자산으로 남겨야 합니다.

### 3. 의사결정의 히스토리 추적

하나의 의사결정이 하나의 문서로 독립되어 남기 때문에 시간이 지나도 당시의 고민을 그대로 복기할 수 있습니다.

기존 문서를 덮어쓰는 대신 새로운 ADR을 차곡차곡 쌓는 방식 덕분에 의사결정의 히스토리를 자연스럽게 추적할 수 있게 됩니다.

## ADR 실무 적용 가이드

### 1. 언제 ADR을 작성해야 할까?

ADR은 모든 결정에 작성할 필요가 없습니다. 다음과 같은 경우에 작성하는 것을 추천합니다:

- **아키텍처 변경**: 모놀리스에서 마이크로서비스로의 이전, 레이어 구조 변경 등
- **기술 스택 변경**: 데이터베이스 교체, 언어/프레임워크 변경, 메시지 브로커 도입 등
- **대규모 리팩토링**: 성능 개선, 보안 강화, 확장성 확보를 위한 구조적 변경
- **도메인 복잡도에 따른 기술적 타협**: 시간 제약, 리소스 제한으로 인한 비효율적인 결정

### 2. ADR 템플릿

다음은 Backend 개발자에게 추천하는 ADR 템플릿입니다:

```markdown
# ADR-001: [제목]

## Status
Proposed | Accepted | Deprecated | Superseded by [ADR-XXX]

## Context
어떤 문제 상황에서 논의가 시작되었는가?

- 현재 시스템 상황
- 문제점
- 영향 범위
- 제약 조건

## Decision
여러 선택지 중 무엇을 선택했고, 그 이유는 무엇인가?

- 선택된 옵션
- 대안들과 비교
- 선택 이유

## Consequences
그 결정으로 얻는 이득과 감수해야 할 비용은 무엇인가?

### 긍정적 결과
- 이득 1
- 이득 2

### 부정적 결과
- 비용 1
- 비용 2

## Retrospective
진행하며 얻은 인사이트나 교훈은 무엇인가?

## References
관련 링크, 문서, 토론 등
```

### 3. ADR 관리 방법

- **저장소**: Git 레포지토리 내 `docs/adr/` 폴더에 저장
- **파일 네이밍**: `YYYY-MM-DD-adr-001-title.md`
- **번호 부여**: 순차적으로 번호 부여 (ADR-001, ADR-002, ...)
- **상태 관리**: Proposed → Accepted → Deprecated → Superseded

## 실무 예시

### ADR-001: Supabase PostgreSQL 도입

```markdown
# ADR-001: Supabase PostgreSQL 도입

## Status
Accepted

## Context
Stock Analyzer Backend 프로젝트에서 데이터베이스를 선택해야 합니다.

요구사항:
- 실시간 데이터 수집 및 분석
- 트래픽 급증에 대응하는 확장성
- 개발 생산성 높은 도구

현재 상황:
- 초기 단계로 빠른 프로토타이핑 필요
- 인력 제약으로 복잡한 인프라 관리 어려움

## Decision
Supabase (PostgreSQL 기반)를 사용합니다.

### 대안 비교

| 옵션 | 장점 | 단점 |
|------|------|------|
| 직접 PostgreSQL | 완전한 제어권 | 인프라 관리 부담 |
| AWS RDS | 안정성, 확장성 | 비용, 설정 복잡도 |
| Supabase | 개발 생산성, 실시간 기능 | 벤더 락인 우려 |

### 선택 이유
1. 빠른 개발 시작: 인증, 실시간 구독 기능 내장
2. 적정 비용: 초기 단계에서 비용 효율적
3. 확장성: PostgreSQL 기반으로 필요시 마이그레이션 용이

## Consequences

### 긍정적 결과
- 빠른 프로토타이핑 가능 (MVP 2주 완료)
- 인증/인가 구현 시간 단축
- 실시간 데이터 동기화 쉬움

### 부정적 결과
- 벤더 락인 우려: 향후 마이그레이션 필요할 수 있음
- 제한된 커스터마이징: 일부 고급 기능 사용 제한

## Retrospective
3개월 운영 결과:
- 초기 빠른 개발에 큰 기여
- 트래픽 증가 시 비용 상승 우려 → 비용 최적화 방안 고려 필요

## References
- [Supabase 공식 문서](https://supabase.com/docs)
- [PostgreSQL vs MySQL 비교](https://example.com)
```

## ADR 문화 도입을 위한 팁

### 1. 완벽함을 추구하지 마세요

ADR은 완벽한 문서가 아닙니다. "내가 왜 이런 선택을 했지?"라는 질문에 답할 수 있으면 충분합니다. 짧더라도 작성하는 게 아예 안 쓰는 것보다 낫습니다.

### 2. 자동화와 결합하세요

PR을 생성할 때 ADR 번호를 참조하게 하거나, ADR 변경 시 자동으로 팀 Slack에 알림을 보내는 등 자동화와 결합하면 효율적입니다.

### 3. 회고 시간에 ADR을 검토하세요

정기적인 회고 시간에 최근 ADR들을 검토하고, 데프리케이트된 ADR을 정리하거나 새로운 인사이트를 추가하세요.

### 4. 새로운 팀원 onboarding에 활용하세요

새로운 팀원이 들어왔을 때 ADR 목록을 읽어보게 하면 시스템의 의사결정 히스토리를 빠르게 파악할 수 있습니다.

## 마치며

우아한형제들 블로그에서 언급한 것처럼:

> 기록은 더 이상 의무적으로 작성하는 귀찮은 문서가 아니라, 미래의 우리에게 길을 안내하는 팀의 기억 장치가 되었습니다.

Backend 개발자에게 ADR은 코드와 함께 중요한 자산입니다. 시스템의 복잡도가 날로 증가하는 상황에서, 기술적 결정을 체계적으로 관리하는 것은 선택이 아닌 필수입니다.

작은 불편함부터 시작해보는 건 어떨까요? 다음번에 중요한 기술적 결정을 내릴 때, 간단한 ADR부터 작성해보세요. 나중에 그 ADR이 당신의 "자신감 있는 답변"이 되어줄 것입니다.
