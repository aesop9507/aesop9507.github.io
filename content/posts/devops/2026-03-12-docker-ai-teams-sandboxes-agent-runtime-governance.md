AI 코딩 도구가 팀에 들어온 뒤 생산성이 오르는 건 맞지만, 운영 관점에서는 곧바로 새로운 문제가 생깁니다.

- 에이전트가 어디서 실행되는가?
- 어떤 권한으로 파일/네트워크에 접근하는가?
- 실패했을 때 누가 재현하고 추적하는가?
- 실험 비용이 언제부터 통제 불가능해지는가?

Docker Blog의 최신 포스트 **“Building AI Teams: How Docker Sandboxes and Docker Agent Transform Development” (2026-03-11)** 는 이 질문에 대해 꽤 명확한 방향을 제시합니다.

핵심은 단순합니다.

> **한 명의 범용 에이전트에 모든 책임을 몰지 말고, 역할별 에이전트를 격리된 실행 환경에서 협업시키자.**

이 글에서는 해당 아이디어를 DevOps/플랫폼 운영 관점에서 해부해 보겠습니다.

---

## 왜 "멀티 에이전트"가 운영적으로 유리한가

단일 에이전트 방식은 초기 진입이 쉽습니다. 하지만 실제 제품 개발로 넘어가면 다음 문제가 반복됩니다.

1. 컨텍스트 오염: 설계/구현/테스트 문맥이 한 세션에 섞여 품질 저하
2. 권한 과다: 한 에이전트가 모든 파일·명령 접근 권한을 보유
3. 책임 불분명: 실패 원인(설계? 구현? 테스트?)이 로그에서 분리되지 않음

멀티 에이전트 모델(예: PM, Designer, Engineer, QA, Fixer)로 나누면,

- **업무 경계가 명확해지고**
- **권한을 최소 단위로 분리**할 수 있으며
- **장애 분석 단위가 작아져** MTTR(Mean Time To Recovery)을 줄일 수 있습니다.

즉, 단순 "AI가 빨리 짜준다"가 아니라 **운영 가능한 개발 시스템**이 됩니다.

## Docker Sandboxes의 본질: 속도가 아니라 "통제 가능한 실행"

포스트에서 강조되는 Docker Sandboxes의 가치도 결국 격리입니다.

DevOps 관점에서 샌드박스는 다음 4가지를 동시에 만족해야 합니다.

### 1) 실행 격리 (Blast Radius 제한)
에이전트 실수(잘못된 스크립트, 의존성 파손, 과도한 리소스 사용)가 호스트 전체 장애로 번지지 않아야 합니다.

### 2) 재현 가능성 (Reproducibility)
"어제는 됐는데 오늘은 안 됨"을 줄이려면, 실행 환경이 이미지/설정으로 고정되어야 합니다.

### 3) 수명 제어 (Ephemeral Lifecycle)
작업 단위로 생성-폐기되어야 드리프트가 줄고, 장기 오염된 환경을 제거할 수 있습니다.

### 4) 감사 가능성 (Auditability)
누가 어떤 컨텍스트로 어떤 명령을 실행했는지 추적 가능해야 보안·컴플라이언스 대응이 됩니다.

샌드박스는 개발 편의 기능이 아니라, AI 시대의 **기본 인프라 제어면(Control Plane)** 입니다.

## 운영 아키텍처 패턴: "Root Orchestrator + Specialist Agents"

Docker 글의 예시처럼 root agent가 조율하고 하위 에이전트가 역할 수행하는 구조는, DevOps에서 아래처럼 해석할 수 있습니다.

- **Root Agent (Orchestrator)**
  - 요구사항 분해
  - 에이전트 간 작업 순서 결정
  - 승인/중단 게이트 관리

- **Designer Agent**
  - UI/UX 결과물 생성
  - 디자인 토큰/접근성 기준 검증

- **Engineer Agent**
  - 구현 및 리팩터링
  - 빌드/로컬 테스트 실행

- **QA Agent**
  - 테스트 실패 분류
  - 재현 시나리오 생성

- **Fixer Agent**
  - 실패 원인 해결 후 재검증

이 구조의 장점은 "역할 분리"보다도 **파이프라인 분리**에 있습니다.
각 단계가 로그, 아티팩트, 실패코드를 독립적으로 남기므로 자동 회복/재시도 정책을 적용하기 쉬워집니다.

---

## 실무 적용 시 반드시 넣어야 할 거버넌스 5가지

### 1) 권한 프로파일 분리
에이전트별로 파일 경로·네트워크·명령 실행 권한을 다르게 줘야 합니다.

- Engineer: 소스 코드 write 허용
- QA: 테스트 실행 중심, write 최소화
- Root: 직접 구현 금지, 오케스트레이션 전용

### 2) 승인 게이트(Approval Gate)
고위험 액션은 사람 승인 또는 정책 승인 없이는 실행 금지해야 합니다.

- 배포
- 시크릿 접근
- 인프라 변경(terraform apply 등)
- 외부 시스템 쓰기 작업

### 3) 비용 상한(Budget Guardrail)
모델 호출/샌드박스 시간/토큰 사용량에 상한을 두지 않으면 팀 확장 시 비용이 폭증합니다.

### 4) 관측성 표준화
각 에이전트 실행 로그에 최소 메타데이터를 강제하세요.

- task_id / agent_role / start-end time
- model / tool usage / exit status
- artifact path / retry count

### 5) 실패 처리 정책
"실패 = 수동 개입" 구조를 줄이기 위해 자동 정책을 넣어야 합니다.

- 일시 오류: 자동 재시도 1~2회
- 동일 오류 반복: 즉시 blocker 전환
- blocker 전환 시: 필요한 입력(Need) 명시

## 보안 관점에서 보는 멀티 에이전트의 함정

에이전트를 늘리면 생산성은 오르지만 공격면도 커집니다.

대표 리스크는 다음과 같습니다.

1. **프롬프트 인젝션 전파**: 한 에이전트가 오염된 지시를 다른 에이전트로 전달
2. **권한 상승 우회**: 저권한 에이전트가 고권한 에이전트 경유로 우회 실행
3. **비인가 데이터 이동**: 샌드박스 간 민감 정보 복사

대응은 전통적인 제로트러스트 원칙과 같습니다.

- 기본 거부(default deny)
- 최소 권한(least privilege)
- 명시적 승인(explicit approval)
- 전 구간 로깅(full traceability)

결론적으로 "AI 에이전트 운영"은 새로운 영역 같지만, 보안 원칙은 오히려 더 전통적이어야 합니다.

## 도입 로드맵 (90일)

### 0~30일: 안전한 파일럿
- 한 팀/한 서비스에서만 시범 운영
- 읽기 중심 권한으로 시작
- 승인 게이트 강제

### 31~60일: 역할 분리 고도화
- 단일 에이전트 → 역할별 에이전트 전환
- QA/Fixer 분리
- 실패 유형별 재시도 정책 적용

### 61~90일: 조직 표준화
- 에이전트 템플릿(역할·권한·로그 스키마) 배포
- 비용 대시보드와 알람 연결
- 보안 리뷰 체크리스트 릴리즈

핵심은 "전사 도입"이 아니라 **통제 가능한 속도로 확장**하는 것입니다.

## 마무리

Docker의 이번 글이 주는 메시지는 명확합니다.

- AI 에이전트는 이제 "개인 생산성 도구"를 넘어,
- 팀 단위 소프트웨어 전달 체계를 재구성하는 단계로 이동했습니다.

하지만 성패는 모델 성능이 아니라 운영 설계에서 갈립니다.

- 격리 없는 자동화는 위험하고
- 거버넌스 없는 멀티 에이전트는 금방 혼란을 만듭니다.

따라서 우리가 집중해야 할 질문은 이것입니다.

> "AI가 코드를 잘 쓰는가?"보다, **"AI 팀을 안전하고 재현 가능하게 운영할 수 있는가?"**

그 기준에서 보면 Docker Agent + Sandboxes 접근은, DevOps 팀이 바로 실험해볼 가치가 충분한 방향입니다.
