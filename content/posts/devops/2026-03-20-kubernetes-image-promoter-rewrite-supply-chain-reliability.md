Kubernetes 생태계에서 `registry.k8s.io` 이미지는 사실상 전 세계 클러스터의 기반 공급망입니다. 이 경로가 흔들리면 릴리스 자체가 멈춥니다.

Kubernetes Blog(2026-03-17) **"The Invisible Rewrite: Modernizing the Kubernetes Image Promoter"**는 이 핵심 파이프라인(kpromo)을 사용자 체감 변경 없이 전면 재작성한 사례를 다룹니다. DevOps 관점에서 이 글이 중요한 이유는 단순한 "Go 리팩터링"이 아니라, **신뢰성과 보안 보증을 잃지 않고 운영 중 핵심 경로를 교체하는 방법론**을 보여주기 때문입니다.

**출처:** https://kubernetes.io/blog/2026/03/17/image-promoter-rewrite/

---

## 왜 이 리라이트가 어려웠는가

kpromo는 단순 이미지 복사기가 아닙니다.

- staging → production 이미지 승격
- cosign 서명
- 다수 지역 미러로 시그니처 복제
- SLSA provenance attestation 생성/검증

즉, 성능 최적화 한 번 잘못하면 보안·무결성·릴리스 일정이 한 번에 무너질 수 있는 구조입니다.

기존 파이프라인의 병목은 명확했습니다.

1. 프로덕션 승격 잡이 30분 이상 소요
2. 레이트 리밋 충돌로 실패 빈도 증가
3. 모놀리식 코어로 기능 확장(검증/스캔/증적) 난이도 상승

핵심은 "더 빠르게"가 아니라 **"실패 확률을 낮추면서 더 빠르게"** 였습니다.

---

## 아키텍처 전환 포인트: Monolith → Phase Pipeline

리라이트 후 파이프라인은 아래 7단계로 분리됐습니다.

1. Setup
2. Plan
3. Provenance
4. Validate
5. Promote
6. Sign
7. Attest

이 구조의 실무적 의미는 큽니다.

### 1) 장애 격리 단위가 생긴다
기존 단일 함수형 흐름에서는 실패 원인이 얽혔지만, 단계 분리 후에는 "어느 단계에서 깨졌는지"가 즉시 식별됩니다.

### 2) 레이트 리밋 예산을 단계별로 독점 사용
병렬성 자체보다 중요한 건 **경합 제거**입니다. 특히 서명(Sign)과 시그니처 복제(Replication)를 분리해 레이트 리밋 충돌을 줄인 점이 핵심 개선입니다.

### 3) 보안 검증을 '부가 기능'이 아닌 필수 단계로 승격
Provenance/Validate/Attest가 명시적 단계가 되면서, 보안·증적 흐름이 성능 튜닝에 묻히지 않게 됩니다.

## 성능 개선에서 배울 점: 단순 병렬화보다 '불필요한 일 제거'

원문에서 특히 인상적인 부분은 병렬화 자체보다 **작업량 감축 전략**입니다.

- Plan 단계 registry read 병렬화로 약 20분 → 약 2분
- 2단계 태그 조회로 API 호출 대폭 감축
- 소스 시그니처 존재 여부를 먼저 확인해 미러 순회 작업 축소
- 요청별 timeout + retry로 장시간 hang 방지
- HTTP 연결 재사용으로 인증 핸드셰이크 낭비 감소

DevOps 운영에서 자주 놓치는 포인트는 "CPU를 더 쓰면 빨라진다"가 아니라,
**"안 해도 되는 API 호출을 없애면 훨씬 안정적이고 싸게 빨라진다"** 는 점입니다.

---

## 보안 공급망 관점: SLSA/서명/증적을 운영 경로에 내장하기

이번 변경은 단지 속도 향상이 아닙니다.

- SLSA provenance 검증/생성 기본 경로화
- cosign 기반 서명 체계 고도화
- in-toto predicate 타입 명시로 증적 구조 명확화

이 흐름은 "사후 감사용 로그"가 아니라, **릴리스 승인 조건 자체를 기계적으로 강제**하는 방향입니다.

플랫폼팀에 적용하면 아래 정책으로 치환할 수 있습니다.

- "이미지 배포"가 아니라 "검증된 provenance 포함 이미지 배포"
- "스캔 리포트 첨부"가 아니라 "스캔 통과 전 Promote 불가"
- "사람 판단"이 아니라 "파이프라인 단계 게이트"

## 무중단 리라이트 전략: 사용자 인터페이스 불변성 유지

가장 실전적인 교훈은 이것입니다.

> 내부는 크게 바꿔도, 사용자 계약(플래그/매니페스트/워크플로)은 유지하라.

kpromo 팀은 다음 계약을 유지했습니다.

- CLI 플래그 동일
- YAML manifest 형식 동일
- 기존 Prow job 연속 동작

그래서 "대규모 변경"을 했지만 사용자에게는 "아무 일도 없던 것처럼" 보였습니다.

플랫폼 개편 시 권장되는 3단계 패턴도 명확합니다.

1. 신구 경로 병행(새 엔진 기본 전환 전 soak)
2. 단계별 릴리스(phase-by-phase)
3. 레거시 완전 제거는 안정성 검증 후 마지막에

## 운영 지표로 재해석한 이번 성과

원문 수치를 운영 지표 관점으로 읽으면 다음과 같습니다.

- 코드베이스 20% 축소: 유지보수 인지 부하 감소
- 40+ PR / 3개 릴리스: 작은 변경 단위로 위험 분산
- 19개 장기 이슈 해소: 구조적 부채 상환
- 재시도/타임아웃/백오프 표준화: MTTR 단축 가능성 증가

즉, "한 번의 대수술"이 아니라 **관측 가능한 작은 수술들의 체계적 누적**이었습니다.

## 우리 팀에 바로 적용할 체크리스트

핵심 배포 파이프라인(이미지 승격, artifact publish, helm release 등)을 운영 중이라면 아래를 점검해볼 수 있습니다.

### 아키텍처
- [ ] 모놀리식 실행 흐름을 단계형 파이프라인으로 분해했는가
- [ ] 단계별 실패 원인을 분리 관측 가능한가
- [ ] 보안 검증(서명/증적/스캔)이 명시적 게이트인가

### 신뢰성
- [ ] 요청 단위 timeout이 모두 존재하는가
- [ ] retry/backoff 정책이 호출 종류별로 정의됐는가
- [ ] 레이트 리밋 경합 구간(서명/복제/검증)이 분리됐는가

### 성능
- [ ] 고비용 API 호출 전 '존재성 체크'로 조기 탈락시키는가
- [ ] 병렬화 대상과 직렬화 대상이 명확히 구분됐는가
- [ ] 연결 재사용/인증 캐시로 핸드셰이크 오버헤드를 줄였는가

### 변경 관리
- [ ] 사용자 계약(CLI, 스펙, 입력 포맷) 불변 원칙이 있는가
- [ ] canary/soak 구간 없이 기본값 전환하지 않는가
- [ ] 레거시 제거는 관측 지표 안정화 후 진행하는가

---

## 마무리: 보이지 않는 성공이 진짜 플랫폼 성공

이번 사례의 결론은 단순합니다.

- 눈에 보이는 기능 추가보다
- **보이지 않는 핵심 경로의 신뢰성 개선**이
- 플랫폼 전체 생산성을 더 크게 끌어올린다.

DevOps 팀이 인정받는 순간은 "새 대시보드 출시"보다,
**릴리스가 더 빠르고 더 안전해졌는데 아무도 장애를 체감하지 못할 때**입니다.

kpromo 리라이트는 바로 그 기준을 충족한, 교과서적인 플랫폼 엔지니어링 사례입니다.
