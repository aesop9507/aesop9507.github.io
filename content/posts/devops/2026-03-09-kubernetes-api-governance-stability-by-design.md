## 왜 API Governance를 DevOps가 신경 써야 하는가

인프라 팀은 흔히 "배포 안정성"과 "운영 자동화"를 핵심 KPI로 본다.  
그런데 Kubernetes 환경에서 배포/운영 안정성의 근본에는 생각보다 더 중요한 축이 하나 있다.

**바로 API 안정성(compatibility contract)이다.**

- 컨트롤러가 기대하는 필드 구조
- GitOps 파이프라인이 적용하는 manifest 스키마
- 플랫폼 템플릿(Helm/Kustomize)이 전제하는 기본값

이 중 하나라도 예고 없이 깨지면, 장애는 애플리케이션이 아니라 **플랫폼 경계면(API)** 에서 시작된다.

Kubernetes SIG Architecture의 "API Governance" 인터뷰가 흥미로운 이유는,
이 주제를 단순 문서 규칙이 아니라 **생산 시스템 신뢰성의 본질**로 다루기 때문이다.

---

## 핵심 메시지 1: Kubernetes에서 API는 REST만이 아니다

인터뷰에서 Jordan Liggitt는 "API 표면"을 REST 엔드포인트로 한정하지 않는다.

Kubernetes의 API 표면은 다음까지 포함한다.

- CLI 플래그
- 컴포넌트 설정 파일
- 바이너리 실행 방식
- 백엔드(예: 런타임, 저장소)와의 인터페이스

즉, DevOps 관점에서 보면 우리가 매일 다루는 거의 모든 운영 접점이 API다.

### 실무 함의

1. **릴리스 노트 읽기 범위를 넓혀야 한다**  
   단순 "v1 API 변경"만 보는 습관으로는 부족하다.

2. **플랫폼 변경 영향도 분석 템플릿**이 필요하다  
   "manifest 호환성"뿐 아니라 "플래그/설정/런타임 경계"까지 체크해야 한다.

3. **표준화되지 않은 운영 스크립트**는 잠재 장애원이다  
   내부적으로 굳어진 실행 옵션도 사실상 API 계약이기 때문이다.

---

## 핵심 메시지 2: 안정성과 혁신은 트레이드오프가 아니라 운영 설계 문제

API Governance 팀의 목표는 두 가지를 동시에 달성하는 것이다.

- 안정성 유지 (사용자 계약 보호)
- 변화 허용 (기능 진화)

"아무것도 안 바꾸면 안정적"이라는 접근은 현실적으로 불가능하다.  
반대로 "빨리 바꾸자"만 강조하면 하위 호환성 비용이 사용자에게 전가된다.

Kubernetes가 택한 방식은 다음과 같다.

- API conventions 문서화
- 설계/구현 단계에서의 리뷰 개입
- 린터/자동 검증으로 반복 실수 차단
- 새 케이스가 나오면 규칙 자체를 지속 업데이트

### DevOps 팀이 적용할 수 있는 패턴

- **정책 문서 + 자동 검증 + 리뷰 문화**를 세트로 운영하라.
- 문서만 있으면 지켜지지 않고,
- 자동화만 있으면 맥락을 잃고,
- 리뷰만 있으면 병목이 생긴다.

세 가지가 함께 있을 때만 플랫폼 품질 게이트가 작동한다.

## 핵심 메시지 3: CRD는 혁신의 가속기이자 일관성 리스크였다

인터뷰에서 가장 중요한 포인트는 CRD 이야기다.

CRD 초기에는 "누구나 원하는 API를 바로 만들 수 있다"는 점이 엄청난 혁신이었다.
하지만 동시에 다음 문제가 발생했다.

- 스키마 불일치
- 검증 부족
- 일관성 없는 필드 설계
- 이미 저장된 비정상 데이터 처리 난제

이후 Kubernetes는 CRD 스키마 의무화, 검증 기능 강화, ratcheting validation 등으로
"자유도"와 "안정성" 사이 균형을 다시 맞춰 왔다.

### 플랫폼 엔지니어링 관점의 교훈

CRD를 많이 쓰는 조직일수록 다음을 표준화해야 한다.

1. **CRD 설계 기준서**
   - 필드 네이밍 규칙
   - spec/status 분리
   - default/nullable 정책

2. **버전 전환 전략(v1alpha1 → v1beta1 → v1)**
   - deprecation 공지 주기
   - conversion webhook 테스트
   - 롤백 가능성 검토

3. **데이터 호환성 전략**
   - 기존 객체와 신규 검증의 충돌 시나리오
   - 점진적 강화(ratcheting) 원칙

CRD를 "빠른 확장 도구"로만 보면 장기적으로 운영 비용이 폭증한다.

## API Review 타이밍: 늦게 받을수록 비용이 기하급수로 증가한다

인터뷰에서 반복된 메시지는 명확하다.

- 설계 단계 조기 리뷰가 가장 저렴하다.
- 구현 막바지 리뷰는 구조 변경 비용이 매우 크다.

Kubernetes 릴리스 사이클에서도
- enhancements freeze 전: 설계 논의 집중
- code freeze 전: 구현 정합성 검토 집중
패턴이 나타난다.

### 우리 팀에 맞는 실행 규칙

- 큰 API/플랫폼 변경은 **구현 전 설계 리뷰를 필수 게이트**로 둔다.
- "마감 2~3주 전 몰아 리뷰"를 금지한다.
- 변경 제안서 템플릿에 아래 항목을 강제한다.
  - 향후 확장 경로
  - 하위 호환성 보장 방식
  - 실패 시 복구/롤백 전략

## 사용자 관점 계약: "우리가 미래에 더 똑똑해져도, 사용자를 깨뜨리면 실패"

Jordan Liggitt의 마지막 메시지는 DevOps에도 그대로 적용된다.

> 우리는 미래에 더 많은 것을 알게 되겠지만,
> 그때의 개선이 오늘의 사용자 계약을 깨뜨려서는 안 된다.

운영팀이 자주 빠지는 함정은 이것이다.

- "이 구조가 더 깔끔하니 지금 갈아엎자"
- "기술적으로 맞으니 사용자 변경 비용은 감수해야지"

하지만 플랫폼 신뢰는 "기술적 정답"보다 "계약 일관성"에서 생긴다.

---

## DevOps 실행 체크리스트 (바로 적용 버전)

- [ ] API 변경 제안서에 **Compatibility 섹션** 필수화
- [ ] CRD 스키마 린팅/검증을 CI 게이트에 통합
- [ ] API 리뷰를 Sprint 초반에 배치(마감 직전 금지)
- [ ] Deprecated 필드/버전의 sunset timeline 문서화
- [ ] 릴리스 노트에 "사용자 영향"을 기능 설명보다 먼저 작성
- [ ] 주요 플랫폼 API에 대해 계약 테스트(contract test) 운영

## 마무리

이번 Kubernetes API Governance 스포트라이트는
"좋은 API란 예쁜 설계"가 아니라,

**"시간이 지나도 사용자를 배신하지 않는 운영 약속"**

이라는 점을 다시 확인시켜 준다.

DevOps 팀이 안정성과 속도를 동시에 확보하고 싶다면,
배포 자동화만큼이나 API 거버넌스를 제품처럼 운영해야 한다.

- 문서화하고,
- 자동화하고,
- 조기 리뷰하고,
- 호환성 계약을 끝까지 지키는 것.

이것이 결국 플랫폼 신뢰성과 팀 생산성을 함께 높이는 가장 현실적인 길이다.
