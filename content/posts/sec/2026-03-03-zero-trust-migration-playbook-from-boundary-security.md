## 왜 지금 ‘경계보안 중심 모델’만으로는 부족한가

많은 조직이 여전히 **WAF/IPS + VPN + NAC + IP 기반 방화벽 규칙** 조합을 기본 보안 모델로 사용합니다. 이 모델은 과거 온프레미스 중심 환경에서 효과적이었지만, 오늘날에는 다음과 같은 구조적 한계를 드러냅니다.

- HTTPS 암호화 트래픽 비중 증가로 경계 장비 가시성 저하
- 클라우드/컨테이너 확장으로 정적 정책 관리 복잡도 급증
- 재택/원격 근무 확대로 단말 신뢰 경계가 흐려짐
- 공격자가 유효 계정·정상 도구를 악용(Living-off-the-land)하는 비율 증가

핵심은 명확합니다. **‘네트워크 위치’가 신뢰의 근거가 되던 시대는 끝났고, ‘사용자·단말·행위’의 지속 검증이 기본이 되어야 합니다.**

## 사례 요약: 토스페이먼츠의 4단계 보안 고도화

토스페이먼츠 보안팀은 열악한 초기 환경에서 출발해 약 4년에 걸쳐 다음 4축을 단계적으로 구축했습니다.

1. **경계보안 고도화**: 암호화 트래픽 가시성 확보, 하이브리드(IDC+AWS) 보안 체계 구축
2. **내부망 보안 강화**: IDC(Wazuh) + AWS(GuardDuty) 기반 탐지/대응 이중화
3. **컨테이너 런타임 보안**: Kubernetes/Falco 및 클라우드 런타임 탐지 체계 도입
4. **제로트러스트 업무망**: ZTNA + EDR + 사용자 속성 기반 동적 접근제어

이 사례의 중요한 포인트는 “솔루션 교체”가 아니라, **운영 모델 자체를 단일 방어선에서 다층 방어(Defense in Depth)로 바꿨다**는 점입니다.

## 실무 인사이트 1) ‘가시성’ 없이는 어떤 보안 전략도 작동하지 않는다

경계보안 고도화에서 가장 먼저 해야 할 일은 정책 추가가 아니라 **관측 가능성(Observability) 확보**입니다.

### 체크포인트

- TLS 트래픽 분석이 가능한지 (복호화/메타데이터 분석 전략 포함)
- 온프레미스와 클라우드의 탐지 이벤트가 한 화면에서 상관분석 가능한지
- 공격 이벤트 발생 시, “누가/어디서/무엇을” 단위로 추적 가능한지

### 실무 팁

- 복호화 도입은 성능·프라이버시·규제 이슈와 함께 설계해야 합니다.
- 전면 도입보다 고위험 구간(결제, 관리자, 인증)부터 단계 적용이 안전합니다.
- 가시성 확보 KPI는 “장비 도입 수”가 아니라 **탐지 가능한 트래픽 비율**로 잡아야 합니다.

---

## 실무 인사이트 2) 내부망은 ‘신뢰 구역’이 아니라 ‘침해 가정 구역’으로 봐야 한다

경계를 통과한 공격을 전제로 하면, 내부망 운영의 중심은 예방보다 **탐지 속도와 대응 자동화**가 됩니다.

### 권장 설계

- 호스트/엔드포인트 텔레메트리 수집 표준화
- 이상행위(권한상승, 비정상 프로세스, 민감 파일 접근) 룰 기반 탐지
- 허용 행위 화이트리스트 + 예외 관리 프로세스 분리
- 자동 차단 조건과 수동 검토 조건을 분리해 오탐 비용 최소화

### 자주 발생하는 실패 패턴

- 룰은 많은데 우선순위가 없어 경보 피로(Alert Fatigue)만 증가
- 경보는 뜨지만 자산/계정 맥락이 없어 대응 지연
- PoC 단계의 탐지 룰을 운영환경에 그대로 적용해 오탐 폭증

---

## 실무 인사이트 3) 컨테이너 보안의 핵심은 ‘이미지 스캔’이 아니라 ‘실행 시점 탐지’다

많은 팀이 CI 단계 이미지 취약점 스캔에 집중하지만, 실제 침해는 런타임에서 드러나는 경우가 많습니다.

예: 컨테이너 내 신규 바이너리 실행, 민감 파일 접근, 비정상 부모-자식 프로세스 체인

### 최소 요구 통제

- 런타임 Syscall 기반 이벤트 수집
- Kubernetes 메타데이터(네임스페이스/워크로드/Pod) 연계
- 고위험 행위에 대한 즉시 차단 또는 격리 워크플로우
- SOC/온콜 체계와 연동된 알림 경로 확보

컨테이너 보안은 서버 보안의 하위 개념이 아니라, **클라우드 네이티브 환경의 별도 방어선**으로 봐야 합니다.

---

## 실무 인사이트 4) 제로트러스트는 VPN 대체가 아니라 ‘의사결정 방식’의 전환이다

제로트러스트를 단순히 “VPN 대신 ZTNA”로 이해하면 실패합니다.

핵심 질문은 다음입니다.

- 현재 접속 요청의 **사용자 신원**은 신뢰 가능한가?
- 접속 단말의 **보안 상태**는 정책 기준을 만족하는가?
- 요청 대상 리소스에 대해 **최소 권한**이 적용되는가?
- 접속 이후에도 이상행위를 **지속 검증**하는가?

### 운영 원칙

1. **Identity-first**: IP가 아닌 사용자/그룹/직무 속성 기반 권한
2. **Device posture**: 단말 상태(패치, 보안 에이전트, 위협 상태) 연동
3. **Least privilege**: 네트워크 전체 접근이 아닌 리소스 단위 접근
4. **Continuous verification**: 1회 인증이 아닌 세션 중 재평가

## 우리 조직에 적용하는 90일 실행 로드맵

### 0~30일: 가시성·기준선 확보

- 고위험 업무/시스템 식별 (결제, 관리자, 인증, 고객데이터)
- 현재 접근 통제 모델(VPN/NAC/IP ACL) 리스크 맵 작성
- 내부·클라우드 로그 수집 및 공통 스키마 정리
- 보안 이벤트 심각도 기준(Critical/High/Medium/Low) 재정의

### 31~60일: 다층 방어 파일럿

- 내부망 이상행위 탐지 룰셋 최소형 운영 시작
- 컨테이너 런타임 탐지(신규 바이너리/민감 파일/권한 상승) 도입
- 고위험 계정·시스템에 조건부 접근 정책(Conditional Access) 적용
- 탐지→티켓→조치까지 자동화 체인 1개 이상 구축

### 61~90일: 제로트러스트 확장

- 사용자 속성 기반 동적 권한 관리 적용 범위 확대
- 단말 상태 불충족 시 접근 제한/격리 정책 활성화
- 예외 승인 프로세스(기한·사유·재검토) 정식 운영
- KPI 리뷰: MTTD/MTTR, 오탐률, 미승인 접근 차단율

## 결론

토스페이먼츠 사례가 주는 가장 큰 메시지는 다음 한 문장으로 요약됩니다.

> **보안 성숙도는 ‘장비 수’가 아니라, 침해를 가정한 상태에서 얼마나 빠르게 탐지·판단·차단하느냐로 결정된다.**

경계보안은 여전히 필요합니다. 다만 그것만으로는 부족합니다. 앞으로의 보안 경쟁력은 **하이브리드 가시성, 런타임 탐지, 그리고 제로트러스트 기반의 동적 접근제어**를 얼마나 유기적으로 운영하느냐에 달려 있습니다.
