## 왜 이 이슈가 중요한가

2026년 3월, The Hacker News를 통해 **Laravel 유틸리티로 위장한 Packagist 패키지가 Windows/macOS/Linux 전부에서 동작하는 RAT를 배포**한 정황이 공개됐습니다. 단순 악성코드 유입이 아니라, 개발팀이 일상적으로 신뢰하는 패키지 레지스트리 경로를 악용했다는 점에서 전형적인 **소프트웨어 공급망 공격**입니다.

특히 이번 사례는 다음 특성이 결합되어 위험도가 높습니다.

- **정상 개발 흐름 침투**: `composer install`만으로 감염 경로가 열림
- **간접 감염 구조**: 직접 악성 코드가 없는 패키지도 의존성으로 악성 패키지를 끌어옴
- **운영환경 동작**: 애플리케이션 부팅/오토로드 시점에 RAT가 활성화될 수 있음
- **권한 상속 위험**: 웹앱 프로세스 권한으로 `.env`, DB 자격증명, API 키 접근 가능

## 사건 요약

보도 기준으로 문제 패키지는 아래 3개였습니다.

- `nhattuanbl/lara-helper`
- `nhattuanbl/simple-queue`
- `nhattuanbl/lara-swagger` (직접 악성코드 대신 의존성으로 악성 패키지 설치 유도)

핵심 포인트는 **직접 악성코드가 없는 패키지도 안전하지 않다**는 점입니다. 의존성 그래프 안에 악성 패키지가 숨어 있으면 결과적으로 동일한 감염 결과를 만들 수 있습니다.

## 기술 분석: 공격자가 잘한 4가지

## 1) 의존성 신뢰 사슬 악용
개발자는 보통 “직접 설치한 패키지”를 검토하고, 하위 의존성까지 깊게 보지 못합니다. 공격자는 이 틈을 노려 **간접 의존성으로 페이로드를 유입**시켰습니다.

## 2) 정적 분석 방해
보도된 분석에 따르면 난독화·인코딩·랜덤 식별자 등을 사용해 코드 가독성을 낮췄습니다. 이 방식은 보안팀/개발자가 육안으로 빠르게 이상 여부를 파악하기 어렵게 만듭니다.

## 3) C2 연결 지속성
RAT가 C2 서버에 주기적으로 재접속을 시도하고, 명령 대기 루프를 유지하는 구조는 **탐지 전까지 장기 체류**를 가능하게 합니다.

## 4) 실행 환경 적응력
쉘 실행 시 여러 함수(`popen`, `proc_open`, `exec`, `shell_exec`, `system`, `passthru`)를 순차 시도하는 방식은 일부 하드닝 환경에서도 우회 가능성을 높입니다.

## 보안 관점에서의 본질: “개발 생산성 경로가 공격 경로가 된다”

공급망 공격은 보안 통제가 약한 경로를 찾는 게 아니라, **업무 효율이 가장 높은 경로를 공격 표면으로 바꾸는 전술**입니다.

- 개발자는 설치 속도를 중시
- 빌드 시스템은 자동화 의존
- 운영은 환경변수에 핵심 비밀 저장

이 세 가지가 만나면, 패키지 1개가 **코드 실행 + 비밀 유출 + 내부 확산의 시작점**이 됩니다.

---

## 실전 대응: “이미 설치했을 가능성”을 전제로 시작

아래는 실제 대응에서 바로 적용 가능한 우선순위입니다.

## 1단계: 노출 확인 (Inventory)
- 전체 저장소에서 `composer.json`, `composer.lock` 전수 검색
- 문제 패키지명 및 퍼블리셔(`nhattuanbl/*`) 패턴 매칭
- CI 아티팩트/컨테이너 이미지에 포함 여부 확인

## 2단계: 격리 및 제거 (Containment)
- 감염 의심 서비스의 아웃바운드 트래픽 임시 제한
- 해당 패키지 제거 후 lockfile 재생성
- 프로덕션 배포 전 SCA/SBOM 재검증

## 3단계: 비밀 재발급 (Credential Hygiene)
- `.env` 기반 시크릿(DB 계정, API 토큰, 메시지 브로커 키) 전면 회전
- 클라우드 IAM 액세스 키 즉시 만료/재발급
- 기존 세션 토큰 강제 무효화

## 4단계: 포렌식 및 재침투 차단
- 웹서버/앱 로그에서 비정상 명령 실행 흔적 점검
- C2 지표(`helper.leuleu[.]net:2096`) 관련 DNS/네트워크 로그 역추적
- 이미지/서버별 IOC 확산 탐지

---

## 예방 전략: “패키지 설치를 통제 가능한 배포 행위로 재정의”

## A. 패키지 허용 정책 (Allowlist)
- 조직 승인 패키지 레지스트리/벤더만 설치 허용
- 신규 퍼블리셔 패키지는 기본 차단 후 리뷰 승인

## B. Lockfile 거버넌스
- `composer.lock` 변경을 보안 리뷰 트리거로 연결
- 하위 의존성 신규 유입 시 자동 위험 점수 평가

## C. 빌드 파이프라인 보안 게이트
- SCA + 악성 패키지 탐지 + 라이선스 점검을 빌드 실패 조건으로 통합
- 배포 직전 SBOM 생성/보관 및 릴리즈 아티팩트 서명

## D. 런타임 탐지 강화
- PHP 프로세스의 비정상 아웃바운드 소켓 연결 알림
- 애플리케이션 계정의 쉘 실행 이벤트 상시 모니터링
- 파일 쓰기/권한 변경 이상행위 탐지

## E. 개발팀 교육 포인트
- “스타 수/다운로드 수/이름 유사성”은 신뢰 지표가 아님
- 의존성 트리 깊은 노드까지 보안 검토 필요
- incident 발생 시 코드 수정보다 **비밀 회전이 선행**

---

## 조직 KPI 제안

- **MTTD-Pkg**: 악성 패키지 유입 탐지 시간
- **Time-to-Rotate-Secrets**: 노출 확인 후 비밀 재발급 완료 시간
- **Unreviewed Dependency Rate**: 보안 검토 없는 신규 의존성 비율
- **Runtime Egress Alert Coverage**: 런타임 아웃바운드 탐지 적용 서비스 비율

## 결론

이번 사례는 “오픈소스 패키지 위험”이라는 오래된 경고를 다시 확인시켜 줍니다. 다만 핵심은 공포가 아니라 운영입니다.

보안팀과 개발팀이 함께 아래 3가지를 표준화하면 재발 가능성을 크게 낮출 수 있습니다.

1. **설치 전 통제**: 허용 목록 + 자동 스캔
2. **배포 전 검증**: lockfile/SBOM 기반 승인
3. **설치 후 감시**: 런타임 egress·명령 실행 모니터링

공급망 공격 대응의 성패는 “좋은 도구”보다 **변경 통제 프로세스**에서 갈립니다.
