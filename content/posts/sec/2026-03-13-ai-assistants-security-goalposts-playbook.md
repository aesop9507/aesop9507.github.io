## 왜 지금 ‘보안 기준선(Security Goalposts)’이 이동하는가

AI 어시스턴트(에이전트형 도구)는 더 이상 단순 질의응답 도구가 아니다. 메일함, 캘린더, 코드 저장소, 메신저, 클라우드 API, 로컬 파일시스템까지 연결해 **사용자 대신 실행**한다. 즉, 기존의 “사용자가 클릭해야 위험이 실행된다”는 가정이 깨진다.

최근 보안 업계 관측(예: Krebs on Security의 AI 어시스턴트 리스크 분석)은 공통적으로 다음 변화를 지적한다.

1. **권한 집중화**: 에이전트 하나에 다수 자격증명(API Key, OAuth Token, Bot Token)이 몰린다.
2. **데이터-코드 경계 붕괴**: 자연어 입력이 실행 계획(코드/행동)으로 변환된다.
3. **신뢰 경계 확장**: 내부자 수준 권한을 가진 ‘자동화 주체’가 조직 내에 다수 생긴다.
4. **저숙련 공격자의 생산성 증폭**: 공격 성공률보다 ‘시도 규모’가 급격히 증가한다.

핵심은 간단하다. **위협의 난이도보다 속도와 규모가 먼저 올라간다.**

## 위협 모델 재정의: “에이전트는 편의 기능이 아니라 특권 시스템”

기존 SaaS 위협 모델에 아래 항목을 추가해야 한다.

### 1) 프롬프트 주입 기반 권한 오남용
- 공격자는 티켓 본문, PR 코멘트, 문서, 채팅 메시지 등 ‘보이는 텍스트’에 지시문을 숨긴다.
- 에이전트가 이를 신뢰하면 승인되지 않은 실행(패키지 설치, 비밀 유출, 외부 전송)이 발생한다.

### 2) 에이전트 관리면(Admin/Web UI) 노출
- 인증/네트워크 설정이 느슨하면, 공격자가 설정 파일·토큰·대화 이력을 획득한다.
- 단일 노출로 다중 시스템 횡적 이동이 가능해진다.

### 3) 스킬/플러그인 공급망 오염
- 편의 확장을 위해 설치한 스킬이 실제로는 권한 상승 통로가 된다.
- “신뢰된 자동 업데이트”는 공격자에게 가장 효율적인 배포 채널이다.

### 4) Lateral Movement 2.0 (AI 유도 횡적 이동)
- 침해 이후 공격자는 추가 익스플로잇보다, 이미 신뢰된 에이전트에게 일을 시키는 편이 빠르다.
- 정상 트래픽처럼 보이므로 탐지 난이도가 높다.

## 방어 원칙: 기존 Zero Trust를 ‘Agent Trust’로 확장

## 원칙 A. 최소권한의 세분화 (Per-Action, Per-Tool, Per-Data)

- “에이전트 1개 = 관리자 권한 1개” 구조를 금지한다.
- 도구별/업무별/시간대별로 토큰을 분리한다.
- 읽기·쓰기·실행·배포 권한을 분리하고, 기본값은 읽기 전용으로 둔다.

**실무 체크포인트**
- 토큰 TTL 단축(예: 1~24시간)
- 작업 단위 임시 자격증명(STS) 적용
- 고위험 작업(삭제/배포/권한변경)은 별도 승인 토큰 요구

## 원칙 B. 실행 격리와 네트워크 제약

- 에이전트 런타임을 일반 사용자 세션과 분리한다.
- egress allowlist를 적용해 외부 전송 경로를 제한한다.
- 로컬 파일 접근 경로를 명시적으로 제한(sandbox mount)한다.

**실무 체크포인트**
- 민감 디렉터리(.ssh, keychain, 비밀저장소) 기본 차단
- 메신저/이메일 자동발신은 도메인/수신자 정책 기반 허용
- 웹훅 호출 대상은 신뢰 도메인만 허용

## 원칙 C. 입력 무해화와 실행 전 검증

- 에이전트가 참조하는 텍스트는 모두 잠재적 공격면이다.
- 입력을 “명령 가능 텍스트”와 “참고 텍스트”로 분리해 처리한다.
- 실행 전 단계에서 정책 엔진이 위험 명령을 차단해야 한다.

**실무 체크포인트**
- PR/이슈/문서에서 수집한 텍스트는 기본적으로 non-executable 컨텍스트 처리
- `install`, `curl|bash`, `exfil`, `disable security` 계열 문구 정책 차단
- 고위험 명령은 사람이 최종 승인(4-eyes)

## 원칙 D. 공급망 검증: 스킬은 코드와 동일하게 심사

- 스킬/플러그인 설치를 CI 승인 파이프라인에 편입한다.
- 서명/해시 고정, 게시자 신뢰도, 권한 스코프를 검증한다.
- 자동 업데이트는 canary + 롤백 가능한 경우에만 허용한다.

**실무 체크포인트**
- “게시자 검증 + 권한 diff + 동작 로그 검증” 3단계 통과 후 배포
- 신규 스킬은 격리 환경에서 24시간 관찰 후 승격
- 스킬 별 SBOM/변경이력 추적

## 원칙 E. 관측성(Observability) 우선 설계

- 에이전트 보안은 사전 차단 + 사후 추적이 모두 필요하다.
- “누가/무엇이/어떤 입력으로/어떤 도구를 호출했는지”를 이벤트로 남긴다.

**필수 로그 스키마**
- `actor`(human/agent id)
- `input_source`(email, issue, chat, doc)
- `tool_call`(name, args hash)
- `resource_access`(files, secrets, api)
- `decision`(allow/deny/escalate)
- `approval_chain`(if high-risk)

## 사고 대응(IR) 관점에서 달라지는 점

기존 IR은 “악성 코드 실행 여부” 중심이었다. 에이전트 시대에는 **정상 권한의 악용**을 먼저 의심해야 한다.

### 에이전트 침해 의심 시 즉시 조치
1. 에이전트 토큰 일괄 회수(rotate/revoke)
2. 외부 통신 차단(egress quarantine)
3. 최근 도구 호출·메시지 주입 경로 포렌식
4. 공급망(최근 설치/업데이트 스킬) 무결성 검증
5. 정상 작업으로 위장된 데이터 반출 흔적 점검

### 탐지 규칙 예시
- 야간 시간대 대량 API 호출 + 신규 외부 도메인 접속
- 읽기 전용 에이전트의 쓰기/삭제 시도
- 평소 사용하지 않는 툴 조합의 연쇄 호출
- 승인 없는 고위험 명령 실행 시도

## 30일 도입 로드맵 (보안팀 기준)

### Day 1~7: 가시화
- 에이전트 인벤토리 작성(도구, 권한, 토큰, 데이터 접근 범위)
- 고위험 권한 맵 작성(삭제/배포/외부전송)

### Day 8~14: 통제
- 토큰 분리, 만료 단축, 승인 게이트 적용
- 실행 격리, 네트워크 egress allowlist 적용

### Day 15~21: 공급망
- 스킬 승인 프로세스 신설(서명·권한 diff·canary)
- 자동 업데이트 정책 재정의

### Day 22~30: 탐지·훈련
- 에이전트 전용 탐지 룰 배포
- 프롬프트 주입 모의훈련(레드팀 시나리오)
- IR 플레이북 개정 및 런북 자동화

## 결론

AI 어시스턴트는 생산성을 올리는 도구이지만, 보안 관점에서는 **고권한 자동화 시스템**이다. 조직이 해야 할 일은 도입을 멈추는 것이 아니라, 다음을 기본값으로 만드는 것이다.

- 최소권한 + 격리
- 입력 무해화 + 실행 검증
- 공급망 심사 + 롤백 체계
- 전구간 감사로그 + 대응 자동화

이 네 가지를 갖추면 “AI 때문에 위험해졌다”가 아니라, “AI를 안전하게 운영할 수 있는 조직”으로 전환할 수 있다.
