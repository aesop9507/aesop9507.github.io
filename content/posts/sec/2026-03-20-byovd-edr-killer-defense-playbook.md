## 왜 이 주제가 중요한가

최근 The Hacker News가 소개한 ESET 분석에 따르면, 확인된 EDR Killer 도구 약 90개 중 54개가 **BYOVD(Bring Your Own Vulnerable Driver)** 전술을 사용했고, 이 과정에서 **34개의 취약하지만 정상 서명된 드라이버**가 악용됐다.

핵심은 단순하다.

- 공격자는 “악성 드라이버”를 직접 서명해 배포하지 않는다.
- 대신 이미 서명되어 신뢰받는 드라이버 중 취약한 것을 가져와 로드한다.
- 그 뒤 커널 권한(Ring 0)으로 EDR/AV를 비활성화하거나 종료한다.
- 마지막 단계에서 랜섬웨어 본체를 실행해 탐지 가능성을 낮춘다.

즉, **암호화기(encryptor)를 숨기는 게임에서, 방어체계를 먼저 끄는 게임으로 무게중심이 이동**하고 있다.

---

## 공격 구조: “암호화 전에 눈을 멀게 한다”

EDR Killer의 운영 목적은 랜섬웨어 본체를 정교하게 은닉하는 것이 아니다. 오히려 다음처럼 역할 분리를 통해 운영 효율을 높인다.

1. **EDR Killer(선행 모듈)**
   - 보안 프로세스 종료
   - 서비스 중지
   - 커널 콜백/보호기능 간섭
2. **Encryptor(후행 모듈)**
   - 대량 파일 암호화
   - 노트 드롭 및 데이터 유출 협박

이 구조는 RaaS(Ransomware-as-a-Service) 생태계와 궁합이 좋다.

- 제휴 공격자는 검증된 “보안 무력화 도구”를 먼저 사용
- 본체는 빌드만 바꿔 반복 사용
- 탐지 우회 비용을 “암호화기”가 아니라 “방어 무력화 단계”에 집중

결과적으로 조직 입장에서는 “암호화 시그니처 탐지”만으로는 늦다. **암호화 이전의 커널/서비스 조작 신호를 먼저 잡아야 한다.**

---

## BYOVD가 계속 통하는 이유

### 1) 신뢰 체인의 역이용
운영체제는 서명 드라이버를 기본적으로 신뢰한다. 공격자는 이 점을 악용해 “합법적 외형”을 얻는다.

### 2) 커널 권한의 파급력
Ring 0 확보 시 사용자 모드 보안 에이전트보다 높은 권한에서 동작 가능하다. 보호 종료, 콜백 조작, 은폐가 쉬워진다.

### 3) 범죄 생태계의 상품화
EDR Killer가 도구/서비스 형태로 유통되면서 공격 진입장벽이 낮아졌다. 공격자는 연구보다 구매를 선택할 수 있다.

### 4) 방어 관성
많은 조직이 여전히 “서명됨 = 안전함” 가정에 머문다. 그러나 현재 위협 모델에서는 **서명 여부보다 취약성·허용정책·행위 통제**가 더 중요하다.

## 실무 대응 플레이북 (우선순위 기반)

## P0: 커널 로드 통제(Prevent)

1. **취약 드라이버 차단 정책 활성화**
   - OS/EDR의 취약 드라이버 블록리스트 기능을 강제 적용
   - 예외 승인 워크플로우를 보안팀 단일 창구로 제한
2. **드라이버 설치 권한 최소화**
   - 로컬 관리자 권한 상시 보유 계정 제거
   - 드라이버 로드 가능한 운영 계정 수를 최소화
3. **애플리케이션/드라이버 허용목록 기반 통제**
   - “서명됨”이 아니라 “허용된 공급망·해시·버전” 기준으로 허용

> 목표: 공격자가 BYOVD 체인에 필요한 “드라이버 로드” 자체를 실패하게 만들기

## P1: 조기 탐지(Detect)

다음 신호를 고신뢰 탐지 룰로 운영한다.

- 비정상 드라이버 로드 이벤트(특히 알려진 취약 드라이버군)
- 보안 제품 서비스 중지/삭제 시도 (`sc stop`, `sc delete`, `taskkill` 연쇄)
- 단기간 내 대량 보안 프로세스 종료 패턴
- Safe Mode 전환 후 보안 서비스 비기동 상태에서의 후속 실행
- EDR 에이전트 outbound 차단/필터링 시도

탐지 설계 팁:

- 단일 IOC보다 **행위 연쇄(Driver Load → Security Service Tamper → Encryption Burst)** 를 룰로 모델링
- “정상 관리자 도구” 악용을 전제로 베이스라인 초과 기반 이상탐지 병행

## P2: 침해 확산 차단(Contain)

1. EDR 침묵(host silence) 징후 시 해당 호스트 자동 격리
2. 동일 드라이버 해시 사용 자산 전수 검색
3. 고권한 계정 세션 강제 만료
4. SMB/RDP/원격관리 경로 임시 제한으로 측면이동 차단

## P3: 복원 및 재발 방지(Recover)

- 골든 이미지 재검증(드라이버/커널 모듈 무결성)
- 보안 에이전트 재배포 전, 로드된 커널 객체 정리 확인
- 예외 허용 정책 사후 감사(누가, 왜, 언제 허용했는지)
- “패치율”이 아니라 “재침투율/재무력화율” 중심 KPI 전환

## 보안팀 운영 관점의 핵심 교훈

### 교훈 1) EDR 단일 의존은 위험하다
EDR이 꺼지는 순간을 가정해, 네트워크·IAM·백업·격리 자동화가 연동되어야 한다.

### 교훈 2) 커널 영역은 별도 통제면으로 봐야 한다
엔드포인트 보안을 앱 레벨만으로 정의하면 늦는다. 드라이버 정책과 커널 텔레메트리 운영이 필수다.

### 교훈 3) “마지막 10분”을 지켜야 랜섬웨어를 막는다
EDR Killer는 보통 암호화 직전에 실행된다. 이 짧은 구간에서 자동 격리/차단이 발동되어야 피해를 줄일 수 있다.

## 7일 내 실행 가능한 체크리스트

- [ ] 취약 드라이버 블록리스트 적용 상태 전 자산 점검
- [ ] 드라이버 로드 이벤트 수집/보존(최소 30일) 확인
- [ ] 보안 서비스 강제 종료 시도 탐지 룰 튜닝
- [ ] EDR agent heartbeat 중단 시 자동 격리 정책 검증
- [ ] 랜섬웨어 테이블탑 훈련에 BYOVD 시나리오 추가
- [ ] 고권한 계정/툴의 드라이버 관련 권한 재검토

## 결론

BYOVD 기반 EDR Killer 확산은 “백신이 탐지하느냐”의 문제가 아니라, **운영체제 신뢰모델을 공격자가 어떻게 우회하느냐**의 문제다. 

방어 전략도 그에 맞게 바뀌어야 한다.

- 서명 신뢰 중심 → 취약성·허용정책 중심
- 단일 시그니처 탐지 → 행위 연쇄 탐지
- 사후 복구 중심 → 암호화 직전 자동 차단 중심

지금 필요한 건 복잡한 신규 도입보다, 이미 가진 통제를 BYOVD 관점으로 재배치하는 일이다. 그 재배치가 빠를수록, 랜섬웨어가 요구하는 “결정적 10분”을 빼앗을 수 있다.
