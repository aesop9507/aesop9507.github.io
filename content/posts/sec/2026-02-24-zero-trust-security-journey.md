## 들어가며

토스페이먼츠는 4년 전 기존 시스템 인수 당시 매우 열악한 보안 환경에서 출발했습니다. 암호화된 트래픽조차 분석할 수 없었고, 단일 방어선에 의존하는 구조였습니다.

이 글은 IDC와 AWS 하이브리드 환경에서 경계보안부터 제로트러스트까지 단계별로 보안 체계를 구축한 여정을 담고 있습니다.

## 우리가 풀어야 했던 4가지 보안 과제

서비스 보안과 단말 보안을 모두 강화하는 포괄적인 보안 체계를 만드는 것이 목표였습니다.

| 영역 | 목표 |
|------|------|
| **경계보안 고도화** | 암호화 트래픽 가시성 확보, 외부 공격 차단 강화 |
| **서버단 내부망 보안** | 경계를 뚫고 들어온 공격에 대비한 내부 방어 |
| **컨테이너 런타임 보안** | 컨테이너 실행 환경 보호 |
| **제로트러스트 업무망 보안** | 사용자와 단말에 대한 지속적 검증, 동적 접근제어 |

단일 방어선의 한계를 인식하고 **체계적인 다층 방어(Defense in Depth)** 전략을 수립했습니다.

## 1. 경계보안 고도화 - 암호화 트래픽 방어, 하이브리드 보안 체계 구축

### 기존 인프라의 한계

기존 인프라 인수 당시 IDC 환경은 IPS(침입방지시스템)와 WAF(웹방화벽)만으로 방어하고 있었습니다.

**가장 큰 문제**: 암호화된 트래픽을 전혀 분석할 수 없었다는 점입니다. 요즘 웹 트래픽의 대부분은 HTTPS로 암호화되어 있습니다. 공격자들도 이를 잘 알고 있죠. 암호화된 트래픽 속에 악성 페이로드를 숨겨 보내면, 기존 보안 장비는 이를 탐지할 수 없었습니다.

### 하이브리드 보안 아키텍처 구축

기존 IDC 환경에 AWS 클라우드를 추가하여 하이브리드 보안 체계를 완성했습니다.

```
┌─────────────────┐     ┌─────────────────┐
│   IDC 환경       │     │   AWS 환경       │
│  (온프레미스)    │     │   (클라우드)     │
├─────────────────┤     ├─────────────────┤
│ • IPS/WAF       │     │ • GuardDuty     │
│ • 트래픽 복호화   │     │ • Shield        │
│ • 가맹점 연동    │     │ • WAF           │
└─────────────────┘     └─────────────────┘
```

두 환경 모두에서 암호화 트래픽 분석 기능을 강화하여, 기존에 탐지하기 어려웠던 암호화된 공격까지 효과적으로 차단할 수 있게 되었습니다.

### 가맹점 공격 인입 대응: 함께 성장하는 보안

토스페이먼츠는 수많은 가맹점과 연결되어 있습니다. 가맹점의 보안이 취약하다면 그 가맹점을 통해 악성 트래픽이 유입될 수 있습니다.

단순히 차단하는 것을 넘어, **가맹점과 함께 보안 수준을 높이는 프로세스**를 구축했습니다:

- "어떤 공격이 발생했고, 어떻게 해결해야 하는지"까지 상세히 안내
- 단순한 서비스 제공자를 넘어 신뢰할 수 있는 보안 파트너로서의 역할

## 2. 서버단 내부망 보안 - "경계를 넘어온 공격에 대비하기"

### 경계보안만으로는 충분하지 않다

정교한 공격자들은 경계보안을 우회하여 내부망으로 침투할 수 있습니다. 제로데이 취약점 공격, 피싱을 통한 내부 침투, 공급망 공격 등 경계를 우회하는 방법은 많습니다.

### IDC 환경: Wazuh 통합 모니터링

**Wazuh**는 오픈소스 보안 플랫폼으로 서버·엔드포인트·클라우드 환경 전반의 보안 기능을 제공합니다.

| 기능 | 설명 |
|------|------|
| 로그 수집 | 서버·엔드포인트·클라우드 로그 통합 |
| 위협 탐지 | 실시간 이상행위 탐지 |
| 취약점 관리 | 자동 취약점 스캔 |
| 규정 준수 | 컴플라이언스 모니터링 |

### AWS 환경: GuardDuty 지능형 탐지

**GuardDuty**는 AWS의 지능형 위협 탐지 서비스입니다.

**실제 탐지 사례**:

```yaml
[이벤트] 일반 계정에서 루트 권한 획득 탐지
  이벤트 유형: PrivilegeEscalation:Runtime/ElevationToRoot
  프로세스: /usr/bin/passwd 실행
  결과: 자동화된 조치 트리거 → 보안팀 알림
```

## 3. 컨테이너 런타임 보안 - "최후의 방어선"

### 왜 컨테이너 보안이 중요한가

컨테이너 환경은 서버와는 다른 특성을 가지고 있습니다:
- 빠르게 생성되고 삭제됨
- 오케스트레이션 플랫폼으로 관리됨
- 전통적인 서버 보안 도구만으로는 동적 특성을 따라잡기 어려움

**컨테이너 런타임 보안은 마지막 방어선**입니다. 경계 보안과 서버 보안을 우회한 공격이라도, 컨테이너 실행 환경에서 이상 행동을 탐지하면 차단할 수 있습니다.

### IDC 환경: Falco 런타임 모니터링

**Falco**는 CNCF(Cloud Native Computing Foundation)에 속한 오픈소스 런타임 보안 도구입니다. Kubernetes, Linux 서버에서 실시간으로 시스템 호출(Syscall)을 감시해 비정상적·의심스러운 행동을 탐지합니다.

```
실시간 Syscall 감시
    ↓
비정상 행위 탐지
    ↓
Falco Sidekick → 다양한 시스템 전파
```

**실제 탐지 사례**:

```yaml
# 민감한 파일 접근 탐지
Alert: Read Sensitive file untrusted
proc.cmdline: cat /etc/shadow
container.name: security-test-container

# 신규 바이너리 실행 탐지
Alert: Drop and execute new binary in container
proc.exec: /tmp/java
```

### AWS 환경: GuardDuty 컨테이너 런타임 보안

AWS 환경에서는 GuardDuty의 컨테이너 보안 기능을 활용합니다.

## 4. 제로트러스트 업무망 보안 - "신뢰하지 않고, 항상 검증한다"

### 기존 업무망 보안 모델의 한계

```
일반 모델:
PC 백신 → NAC (MAC 주소) → VPN → 방화벽 (IP 기반)

문제점:
❌ 백신 검증 우회 가능
❌ IP 기반 접근 통제의 한계 (시간이 지날수록 허용 범위 넓어짐)
❌ VPN 연결 시 내부 네트워크 노출
❌ 알려진 위협만 탐지 (Zero-day 불가)
```

토스페이먼츠의 내부 모의해킹 결과, 일반적인 업무망 보안 모델 구조는 생각보다 쉽게 무너질 수 있다는 것을 확인했습니다.

### 토스페이먼츠의 제로트러스트 보안

**핵심 원칙**: "어떤 것도 신뢰하지 않고, 항상 검증한다"

```
제로트러스트 모델:
모든 PC → ZTNA 연결 → EDR 실시간 탐지

핵심 특징:
✅ IP가 아닌 사용자·속성 기반 동적 그룹
✅ 모든 통신을 네트워크 접속 전에 검증
✅ PC 보안 상태 지속 검증
✅ EDR로 알려지지 않은 위협까지 실시간 탐지·대응
```

### 동적 권한 관리

IP 기반이 아닌 **사용자 속성 기반의 동적 그룹**으로 권한을 관리합니다:

```yaml
# 사용자 속성 기반 권한
user:
  team: "개발팀"
  role: "시니어"

자동 권한:
  - 개발 서버 접근: ✅
  - 프로덕션 DB: ❌ (별도 승인 필요)

# 팀 변경 시 자동 권한 조정
```

직무나 팀이 바뀌면 권한도 자동으로 변경됩니다. 이를 통해 **최소권한원칙**을 보장합니다.

## 배운 점

| 교훈 | 설명 |
|------|------|
| **다층 방어 (Defense in Depth)** | 단일 방어선은 언제든 뚫릴 수 있다 |
| **가시성이 핵심** | 보이지 않는 것은 보호할 수 없다 |
| **자동화 필수** | 사람이 24시간 감시할 수 없다 |
| **제로트러스트** | "신뢰하지 않고, 항상 검증한다" |

## 기술 스택 요약

| 환경 | 영역 | 도구 |
|------|------|------|
| IDC | 경계보안 | IPS, WAF, SSL/TLS 복호화 |
| IDC | 서버 보안 | Wazuh |
| IDC | 컨테이너 보안 | Falco + Falco Sidekick |
| AWS | 서버/컨테이너 보안 | GuardDuty |
| 업무망 | 접근 제어 | ZTNA + EDR |
