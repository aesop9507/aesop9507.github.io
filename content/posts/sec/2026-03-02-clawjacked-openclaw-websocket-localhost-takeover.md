## 왜 이 이슈가 중요한가

ClawJacked는 “인터넷에 직접 노출된 서버”만의 문제가 아닙니다.
핵심은 **개발자 로컬 환경(localhost)** 에서 실행 중인 AI 에이전트 게이트웨이가 브라우저의 WebSocket 연결 특성과 결합될 때 발생하는 신뢰 붕괴입니다.

많은 팀이 다음과 같이 생각합니다.

- `127.0.0.1`에만 바인딩했으니 안전하다
- 비밀번호가 있으니 충분하다
- 내부 툴이니까 사용자 UX를 우선해 자동 승인해도 괜찮다

이번 사례는 이 전제가 동시에 깨질 수 있음을 보여줍니다.

---

## 공격 흐름 요약 (ClawJacked)

보도된 시나리오를 단순화하면 다음과 같습니다.

1. 사용자가 공격자 웹페이지에 접속
2. 페이지 내 JavaScript가 localhost OpenClaw Gateway로 WebSocket 연결 시도
3. 인증 시도에 rate limit이 약하거나 없으면 비밀번호 추측 공격 가능
4. 인증 성공 후 신뢰 디바이스 자동 등록/승인 악용
5. 에이전트 권한으로 설정 조회, 로그 열람, 연결 노드 탐색 등 실행

즉, **“로컬에서만 접근 가능”** 하다는 점이 오히려 공격 표면이 됩니다.

브라우저는 사용자가 방문한 페이지의 JavaScript 실행을 허용하고,
WebSocket은 일반 CORS 모델과 다른 방식으로 동작하기 때문에,
로컬 서비스가 이를 명확히 방어하지 않으면 쉽게 경계가 무너집니다.

## 기술적으로 무엇이 문제였나

### 1) localhost를 신뢰 경계로 본 설계

`localhost == 안전`이라는 가정이 가장 큰 구조적 문제입니다.
실제로는 “사용자 브라우저에서 실행되는 임의 JS”도 localhost로 접근할 수 있습니다.

### 2) 인증 강도 대비 공격 비용 불균형

비밀번호 기반 인증 자체가 나쁜 것은 아니지만,
추측 시도 방어(rate limiting, 지연, lockout)가 부족하면
공격자 비용이 지나치게 낮아집니다.

### 3) 편의 중심의 자동 승인 로직

“로컬 연결이면 pairing 자동 승인” 같은 UX 최적화는
공격자 입장에서 privilege escalation 경로로 변합니다.

### 4) 에이전트 권한의 고가치 집중

에이전트는 여러 시스템과 연결되어 있고 명령 실행 능력이 있습니다.
따라서 단일 진입점이 뚫리면 blast radius가 매우 커집니다.

## 이 사건이 주는 아키텍처 교훈 4가지

### 1) 로컬 서비스도 Zero Trust로 다뤄야 한다

- “네트워크 위치”가 아닌 “요청 컨텍스트”를 신뢰 기준으로 사용
- Origin/Host/Handshake 검증 실패 시 즉시 차단
- 민감 API는 로컬에서도 별도 권한 단계 요구

### 2) 인증은 비밀값 + 행위 방어를 함께 설계한다

- 시도 횟수 제한 (IP/세션/프로세스 단위)
- 실패 누적 시 지연(backoff) + 임시 차단
- 고위험 액션은 재인증 또는 사용자 확인

### 3) 자동 등록/자동 승인 기능은 기본 OFF

- 신규 장치 등록은 반드시 명시적 사용자 승인
- 시간 제한(one-time approval)과 범위 제한(scope-limited)
- 승인 이력 감사 로그 + 철회 기능 제공

### 4) 에이전트는 “지속적 권한을 가진 비신뢰 실행체”로 취급

- 전용 계정/최소권한 토큰 사용
- 민감 시스템 분리(개인 워크스테이션과 격리)
- 재빌드 가능한 샌드박스 운영(감염 가정)

- [ ] CI/CD에 에이전트 설정 정합성 점검(보안 baseline) 자동화
- [ ] incident runbook에 “에이전트 계정 격리/토큰 폐기/로그 포렌식” 절차 포함

## 결론

ClawJacked의 본질은 특정 제품 버그 하나가 아닙니다.
핵심은 **에이전트 시대의 신뢰 모델 전환**입니다.

이제 보안 질문은 이렇게 바뀌어야 합니다.

> “외부에서 들어오지 못하게 했는가?”가 아니라,
> “내부(로컬)에서 시작한 요청도 신뢰하지 않도록 설계했는가?”

로컬 에이전트는 생산성을 크게 높이지만,
동시에 고권한 자동화 엔진을 워크스테이션에 올려놓는 일입니다.

따라서 운영 원칙은 명확합니다.

- localhost 신뢰 폐기
- 기본 거부(default deny)
- 최소 권한 + 명시적 승인
- 빠른 패치 + 지속 모니터링

이 4가지를 지키는 팀이 에이전트 생산성과 보안을 함께 가져갈 수 있습니다.
